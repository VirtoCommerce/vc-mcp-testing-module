## Слайд 3 — Архитектура: три режима, общий фундамент

Теперь, когда вы увидели что и зачем, покажу, как всё устроено. Три режима тестирования, одна общая основа.

**Интерактивный режим**: инженер вводит команду в IDE → Claude Code CLI → 16 агентов → 10 MCP-серверов → 3 реальных браузера. Лучше всего для спринтового тестирования, отладки, тестирования конкретных тикетов.

**Agent Teams режим**: полностью автономный — `autonomous-orchestrator` создаёт команду через TeamCreate API, управляет токен-бакетом (3 слота браузера + 1 репортер), применяет экспоненциальный откат при сбоях, автоматически создаёт JIRA-тикеты для багов.

**CI Pipeline (планируется)**: GitHub Actions, Docker-контейнер с Claude Agent SDK, headless Chromium и Teams Adaptive Card при завершении прогона.

Все три режима читают один и тот же `test-suites.json`, те же CSV-кейсы, тех же агентов, те же файлы знаний. Улучшение в одном режиме — польза всем трём. И — забегая вперёд — у каждого интерактивного пайплайна есть headless-близнец в `ci/`: к трём режимам тестирования в v3 добавились ещё два пайплайна-близнеца — авто-фикс и мониторинг, о них в конце.

## Слайд 6 — 18 slash-команд: полный QA-цикл

18 команд покрывают весь QA-цикл прямо из IDE.

**Три команды, которые используешь ежедневно**: `/qa-smoke` запускает 12 P0-тестов в 2 параллельных потоках и даёт GO/NO-GO вердикт за ~15 минут. `/qa-test VCST-XXXX` читает JIRA-тикет, отправляет нужных агентов, выполняет все acceptance criteria, заводит баги с evidence и закрывает тикет. `/qa-regression sprint` прогоняет ~88 сьютов через 3 браузера параллельно — sprint-селекция читается из плана JIRA.

**Ещё команды для цикла**: `/qa-bug` воспроизводит баг, собирает HAR + скриншоты, заводит JIRA с severity и root cause. `/qa-status` и `/qa-env-check` — read-only и авто-инвокабельные: агент может звать их без явного разрешения пользователя.

И две новые команды, замыкающие петлю бага: `/qa-fix` — автономный фикс уже заведённого бага, и `/qa-monitoring` — онлайн-мониторинг продакшна. О них в трёх финальных слайдах.

Всего 18 команд. Набираешь slash — получаешь автокомплит. Никаких скриптов, дашбордов, переключений контекста.

## Слайд 7 — Pipeline и Quality команды

Три команды, замыкающие QA-цикл.

`/qa-test-lifecycle` — единый pipeline для синхронизации тест-кейсов с кодом. Наводишь на сьют, домен, тикет, PR, модуль или git diff — система прогоняет scope → sync stale → analyze gaps → generate → review → fix → verify → approve. Больше никаких протухших Steps после PR.

`/qa-design` уникален тем, что это **двойной** аудит — Storybook И сторфронт одновременно. Это ловит баги, которые прячутся в одном из контекстов: hover-состояние, которое работает в изоляции, но ломается при интеграции; кнопка, которая в порядке на реальной странице, но сломана в Storybook.

`/qa-verify-fix VCST-XXXX` закрывает цикл JIRA. Читает тикет, воспроизводит изначальный STR, подтверждает фикс, прогоняет регресс-проверки в затронутой области, переводит тикет.

Эти три команды вместе означают: тесты остаются в синке, дизайн-баги ловятся и в изоляции, и в интеграции, а подтверждённые фиксы уходят в релиз, не теряя регрессов.

## Слайд 8 — Power Feature #1: Топ-5 MCP-серверов

MCP — Model Context Protocol — стандарт, по которому Claude вызывает внешние инструменты. Это руки, глаза и память агента.

**#1 Трио Playwright** — три настоящих браузера, Chrome / Firefox / Edge, изолированные контексты, HAR-захват, скриншоты, полная параллельность. Именно это позволяет агентам работать как настоящие пользователи.

**#2 VirtoOZ** — и это главное обновление. VirtoOZ — наш **главный источник документации Virto Commerce**. 12 узкоспециализированных инструментов покрывают PlatformUserGuide, PlatformDeveloperGuide, StorefrontUserGuide, StorefrontDeveloperGuide, BackendSourceCode, MarketplaceUserGuide, DeploymentGuide, B2BExperts. До VirtoOZ агенты были вынуждены угадывать, как устроен VC. Теперь они читают живую, тематическую документацию через скилл `/vc-docs` — перед тем как писать тест, заводить баг или предлагать фикс.

**#3 Atlassian** — JIRA + Confluence. Читать тикеты, заводить баги с evidence, переводить статусы, замыкать QA-цикл.

## Слайд 9 — Power Feature #2: Бизнес-правила и библиотеки edge-кейсов

Это разница между «прошло/упало» и **пониманием почему**.

Все 16 агентов используют один общий файл `business-logic.md` с тестируемыми BL-* инвариантами: цена никогда не может быть отрицательной, наложение скидок не должно превышать 100%, нельзя купить товар с нулевым остатком, пользователи одной организации не видят данные другой. Каждое — однострочное правило с уникальным ID.

При падении теста агент не пишет «ожидал X, получил Y». Он пишет **«BL-PRICE-001 нарушено — применённый купон загнал нетто-цену ниже $0»**. Это совсем другой разговор с разработчиком. И именно эти BL-инварианты команда Developers обязана сохранять в любом авто-фиксе.

У edge-кейсов свои идентификаторы — ECL-*. **13 общих** e-commerce паттернов плюс **7 VC-специфичных**: истёкшая кредитка, нулевая цена, последний товар зарезервирован в чекауте, конкурентная модификация корзины, нарушения границ ролей. Каждый перекрёстно ссылается на BL-инварианты. Каждый сгенерированный тест-кейс цитирует оба.

Обновил одно правило — каждый агент применяет его во всех будущих тестах. Вот в этом и есть рычаг.

## Слайд 10 — Power Feature #3: Параллельный регресс с Quality Gates

Как регресс реально работает. Три слота-браузера одновременно — Chrome, Firefox, Edge. У каждого свой изолированный контекст с HAR-захватом. Оркестратор ставит оставшиеся сьюты в очередь и подаёт их в слоты по мере освобождения.

Таблица селекций справа: smoke — 2 сьюта / ~140 тестов, critical — 5 P0-сьютов / ~230 тестов, sprint — план-зависимый ~88 сьютов / ~2 700 тестов, full — все 101 сьют / ~3 756 тестов.

Вердикты quality gate справа неизменяемы. **APPROVED**: ≥95% pass rate, ноль P0-багов, можно деплоить. **CONDITIONAL**: только P2/P3 проблемы, деплой с зафиксированными follow-up'ами. **BLOCKED**: любой P0/P1 — никакого деплоя независимо от расписания.

При падении сьюта цепочка ретраев переходит между браузерами — chrome → firefox → edge — до двух ретраев. Так мы восстанавливаемся от flake'ов без ручного вмешательства.

## Слайд 12 — GraphQL xAPI: истина схемы + runner-native тесты

Один из самых важных слайдов для бэкенд-QA. Два knowledge-файла плюс `scripts/graphql-runner.ts` превращают xAPI-тестирование в детерминированные CSV-кейсы со schema-валидацией.

**Workflow обновления схемы**: при изменении xAPI мы перезапускаем интроспекцию против `{BACK_URL}/graphql`, перегенерируем `graphql-schema.md`, и CI сразу ловит любой тест с удалёнными полями. Больше никакого тихого слома при изменении схемы.

**Канонический runner** делает schema validation, подстановку переменных, разрешение `@td()` и сбор evidence. Мы никогда не пишем кастомный JS для выполнения GraphQL CSV-кейсов — одна команда `npx tsx scripts/graphql-runner.ts --case <csv>:<ID>` и есть контракт.

Посмотрите CSV-пример справа. **Steps** используют `[AUTH]`, `[GQL-OP]`, `[GQL-VARS]`, `[GQL-EXEC]`, `[GQL-CAPTURE]`. **Assertions** — `[ERRORS]`, `[NULL]`, `[COUNT]`, `[DATA]`, `[VAR]`. Предикаты поддерживают getByPath, арифметику, cross-path, OR-AND.

Эталонный сьют — `050i-graphql-configurations.csv`, готов к копированию новыми авторами.

*[~80 сек]*

## Слайд 13 — Live-discovery тестовых данных: никаких захардкоженных ID

Это решает самую скучную, но самую частую причину протухания тестов.

Каталоги пересеваются. B2B-организации пересоздаются. Virtual-catalog roots мигрируют между средами. Захардкоженные ID и SKU протухают за один-два спринта.

Четыре слоя данных решают это. **`{{VAR}}`** — env-конфиг из `.env`: URL, креды, store IDs. **`@td(ALIAS.field)`** — именованные сущности, против которых ассертим по имени: «тот самый конфигурируемый лэптоп, под который писался тест», «канонический Skyflow-карт», «организация TechFlow». **`live-discover`** — когда нужна структура, а не точное значение: «первый доступный продукт», «любой активный купон». **`random-data`** — для уникальных входных данных, которые мы не ассертим точно: email регистрации, имена организаций. Дефолтный префикс `AGENT-TEST-*` для чистого teardown.

Антипаттерн — захардкоженный GUID продукта или SKU в `Test_Data`. Литерал, который не `{{VAR}}` или `@td()` — это провал ревью, Dimension 6 из 7-мерного ревью.

## Слайд 14 — Теги тест-runner'а + резолвер @td()

CSV-грамматика для браузерных тестов. `test-runner-tags.md` — каноническая ссылка, на которой говорят оба runner'а — `test-runner-agent` и `autonomous-test-runner`.

**Step-теги**: `[NAV]` для навигации, `[ACT]` для click/type/select, `[WAIT]` для точек синхронизации, `[LOGIN]` для аутентификации, `[SETUP]` для предусловий.

**Assertion-теги**: `[DOM]` для наличия элемента, `[STATE]` для состояния компонента, `[MATH]` для числовых проверок, `[API]` для бэкенд-ассертов, `[CONSOLE]` для консоли браузера.

**Кросс-слойные теги**: `[HAR]` для захвата сетевого трафика, `[SCREEN]` для скриншотов, `[PERF]` для тайминга.

Резолвер `@td()` читает `test-data/aliases.json` плюс CSV-строки в `test-data/`. Каталоги пересеваются, ID меняются — алиас остаётся стабильным. Один CSV-апдейт пробрасывается во все потребители. `validate-td-refs.ts` проверяет все ссылки перед запуском. GraphQL-runner отказывает по неразрешённым токенам на lint-стадии.

Эта грамматика — то, что делает CSV-строку одновременно читаемой человеком и исполняемой машиной.

*[~65 сек]*

## Слайд 16 — /qa-test-lifecycle: тесты всегда в синке с кодом

Команда, которая решает старейшую проблему поддержки тестов — протухание тестов после мерджа PR.

Одна команда на весь жизненный цикл тест-кейсов. Восемь шагов: **scope → sync stale → analyze gaps → generate → review (7-dim) → fix → verify → approve**. У каждого шага чёткие deliverables.

Принимает любой scope. `suite <ID>` — прямое ревью качества. `domain <name>` — аудит покрытия по домену. `VCST-XXXX` — вывод кейсов из acceptance criteria тикета. `PR #NNN` — change-driven sync против pull request'а. `module <name>` — для `vc-module-*` репо. `diff` или `changelog <ver>` — чисто git-driven анализ пробелов.

Что производит: обновлённые Steps и Assertions для протухших кейсов, новые кейсы с BL-* и ECL-* цитатами, типизированные теги, отчёт 7-мерного ревью качества, и live-верификация-гейт перед мерджем.

Эта команда заменила устаревший `/qa-sync-tests`. Делегирует test-management-specialist на авторинг и ревью, qa-testing-expert — на live-верификацию.

Pipeline работает без присмотра. Наводишь на PR — получаешь готовый к approval набор апдейтов тест-кейсов.

## Слайд 17 — /qa-seed-data: полная тестовая среда одной командой

Инфраструктура тестовых данных. `/qa-seed-data` генерирует полную тестовую среду через Postman MCP — каталоги, продукты, цены, B2B-организации, пользователи с ролями — и безопасно сносит всё, когда закончено.

Пять профилей. **`minimal`** — один каталог, один продукт, одна цена, один inventory-запись — для smoke-тестов и проверки связности API. **`catalog`** — 3-уровневое дерево категорий, 5 продуктов, мультивалютные цены — для тестов поиска и просмотра. **`b2b`** — одна организация, три пользователя (admin/buyer/viewer), роли — для RBAC-тестов. **`pricing`** — два прайс-листа (USD + EUR), tier-цены, quantity-breaks. **`full`** — всё вместе.

Каждая сущность с префиксом `AGENT-TEST-*`, поэтому `/qa-seed-data teardown` подчищает чисто, не затрагивая реальные данные. Безопасно для любой QA-среды.

Паттерн важен: выбираешь профиль, Postman MCP строит и запускает коллекцию, данные готовы, тесты прогоняются, teardown зачищает. Никакого ручного доступа к БД не требуется.

## Слайд 18 — Реальный конвейер: замкнутая петля Detect → File → Fix → Verify

Это не одна команда — это **замкнутая петля**. Покажу, как пять команд складываются в один сквозной конвейер.

**① Обнаружение — два входа.** `/qa-test VCST-XXXX` берёт тикет, PR или фичу, отправляет специалистов и проверяет каждый acceptance criterion в **реальных браузерах** — с HAR, скриншотами и консолью. Параллельно `/qa-monitoring` следит за **Azure App Insights**: дедуп по fingerprint, триаж, воспроизведение HIGH-confidence живых ошибок продакшна. Один источник — синтетика перед релизом, другой — реальные сигналы из прода.

**② Устранение — одна цепочка.** Оба входа сходятся на `/qa-bug`: воспроизвести дефект и завести баг в JIRA с HAR, скриншотами и ссылками `BL-*`, severity P0–P3. Дальше `/qa-fix` берёт **уже заведённый** баг и проходит лестницу G0→G7 — reproduce-as-test, минимальный фикс, ревью, CI + SonarCloud — и **открывает PR**. Затем `/qa-verify-fix` воспроизводит исходный STR, подтверждает фикс, прогоняет регресс-проверки и переводит JIRA. **Петля замыкается.**

Два ключевых момента внизу. **Человеческий гейт перед мёржем** — `/qa-fix` заканчивается открытым PR, мёржит человек после ревью; никогда не авто-мёрж. И **замкнутость** — проверенные фиксы уходят в релиз, а новые сигналы продакшна снова входят в петлю через `/qa-monitoring`.

Вот как выглядит весь жизненный цикл дефекта с QA-стороны: обнаружить → завести → починить → верифицировать. Никаких ручных скриптов, копипасты в JIRA, потерянных evidence.

## Слайд 19 — /qa-fix: от заведённого бага до открытого PR — автономно

Новая возможность v3, и, пожалуй, самая амбициозная. `/qa-fix` берёт **уже заведённый** баг в JIRA и сам доводит его до открытого PR. Никакого авто-мёржа — об этом через минуту.

В основе — **лестница из восьми гейтов, G0→G7**. Принцип простой: на каждом гейте либо проходим дальше, либо **STOP**. И ключевой момент дизайна — дешёвые гейты без кода идут первыми, **до клонирования репозитория**, поэтому отклонённые тикеты почти ничего не стоят.

**G0 — отбор по пригодности**: баг должен быть простым, низкорисковым, локализованным, без рефакторинга и без breaking change, с сохранением BL-* инвариантов. Правило: *если сомневаешься — BAIL*. **G1 — маршрутизация в один репозиторий**: ровно один разрешённый репо — `vc-module-*`, `vc-platform` или `vc-frontend`. Никаких догадок.

Дальше идёт код. **G2 — воспроизведение (red)**: новый юнит-тест кодирует шаги воспроизведения и **падает на текущем коде** (xUnit для C#, vitest для vc-frontend). **G3 — корректность фикса (green)**: минимальный диф делает этот тест зелёным, при этом **все существующие тесты остаются зелёными и НЕизменёнными**. **G4 — ревью кода**: `backend-reviewer` проверяет диф. **G5 — build + unit CI + SonarCloud quality gate** должен быть зелёным. **G6 — E2E-верификация после деплоя** силами `qa-backend-expert`, потому что бэкенд в CI статичен и симптом проявляется только после редеплоя.

И **G7 — жёсткий стоп**: система **НИКОГДА не мёржит автоматически**. Защита тройная — deny-лист в харнессе, оркестратор не вызывает merge, и самим агентам запрещён `gh pr merge`. Интерактивный режим открывает обычный PR, CI — черновик.

Главная философия: **чистый BAIL — это успех, а не провал**. Всё неоднозначное, сложное, breaking, кросс-репо или с низкой уверенностью оставляется человеку. Неверный GO тратит время ревьюера и подрывает доверие к пайплайну. И `/qa-fix` — это интерактивный близнец headless-скрипта `ci/run-fix-cycle.ts`: оба используют один реестр репозиториев, одну лестницу гейтов и одну защиту от авто-мёржа.

## Слайд 20 — Команда Developers и навыки разработки

Кто всё это делает? Новая, третья команда — **Developers** — единственная команда с правом записи в код. Она клонирует репозитории, ветвит, коммитит, пушит и открывает PR во внешних репозиториях продуктов VirtoCommerce. QA-агенты остаются read-only в GitHub, поэтому область записи строго изолирована именно здесь.

Два агента, оба на Opus. **`fullstack-backend`** чинит **один** репозиторий `vc-module-*` или `vc-platform` — причём это полный стек: бэкенд на .NET 10 / C# **и** Admin SPA на Angular, который поставляется в том же репозитории. Его цикл: воспроизвести баг как падающий тест → минимальный фикс → открыть PR. Никогда не мёржит.

**`backend-reviewer`** владеет **Gate 4** — ревьюит локальный диф **до** того, как PR открыт: один репозиторий, никаких правок существующих тестов, BL-* сохранены, минимально и идиоматично. Возвращает APPROVE или REQUEST_CHANGES.

Их поддерживают три **навыка разработки**. **`/dotnet-unit-test`** — воспроизводит баг как падающий xUnit-тест (red) и доказывает фикс зелёным; никогда не трогает существующие тесты. **`/dotnet-fix`** — реализует минимальный, идиоматичный фикс на .NET 10 в одном модуле до green, с гейтом build + test. **`/angular-admin`** — чинит Admin SPA модуля; и тут важная деталь: у модульных Admin SPA нет JS-харнеса в репозитории, поэтому red→green доказывается **некоммиченным Node scratch-харнесом** с evidence в теле PR.

И ограничение: **у команды нет браузера**. Это чистый код. E2E-верификация — гейт G6 — делегируется обратно QA-экспертам через `/qa-regression`. Ветка всегда `claude/qa-autofix/VCST-XXXX`, один репозиторий за прогон; кросс-модульное или breaking-изменение → STOP и передача человеку.

## Слайд 21 — /qa-monitoring: следить за продакшном, а не только тестировать

Пятый пайплайн-близнец — и он принципиально другой. Вместо выполнения тест-кейсов `/qa-monitoring` **следит за Azure Application Insights** на предмет живых ошибок. Это переход от «проверим перед релизом» к «увидим, что реально ломается у пользователей».

Конвейер слева направо. **Опрос App Insights** на обоих слоях — сторфронт и платформа. **Дедупликация по fingerprint** — хранилище отпечатков переносится между прогонами, поэтому один и тот же стектрейс не поднимает тревогу дважды. **Триаж новых или растущих сигнатур** силами `monitor-triage-agent`. **Воспроизведение HIGH-confidence багов вживую** через QA-экспертов в реальном браузере. И **черновик баг-репорта с блоком Fix Routing плюс алерт в Teams**.

Критически важная граница: это **только детект и репорт**. Система **никогда не заводит JIRA и никогда не авто-фиксит**. Человек берёт подтверждённые черновики и сам запускает `/qa-bug` → `/qa-fix`. Так замыкается полная петля: мониторинг детектит → `/qa-bug` заводит → `/qa-fix` предлагает PR.

Справа — скрипты и общие ассеты. Headless-близнец — `ci/run-monitor.ts` (`npm run ci:monitor`). Алерты — `ci/notify-teams.ts` с `NOTIFY_MODE=monitor`. Библиотека KQL-проб — в `ci/monitoring/queries/`. Хранилище дедупликации — `.seen-fingerprints.json`, кэшируется между прогонами CI. Отчёты идут в `reports/monitoring/MONITOR-*/` плюс Teams, а GitHub Actions запускает всё это по расписанию.

## Слайд 22 — Роадмап: что дальше

Левая колонка — shipped, правая — planned.

**Shipped**: `/qa-test-lifecycle` для единой синхронизации тест-кейсов. Runner-native GraphQL со schema-валидацией и полной грамматикой `[GQL-*]`. Agent Teams автономный регресс с токен-бакетом и экспоненциальным откатом. И две свежие поставки v3 — **конвейер авто-фикса багов `/qa-fix`** (лестница G0→G7, команда Developers, никакого авто-мёржа) и **онлайн-мониторинг продакшна `/qa-monitoring`** (App Insights, fingerprint-дедуп, триаж → live-репро → черновик + Teams, только детект-и-репорт).

**Q2 2026**: CI/CD GitHub Actions пайплайн — ежедневный smoke и еженедельный full, нотификации в Teams, 90-дневная rolling history. GraphQL Runner v2 со schema-diff алертами и chaining-захватом. Мульти-браузерный CI плюс pixel-diff визуальные baseline для всех 55 Storybook-компонентов.

**Q3 2026**: AI root-cause аналитик, сопоставляющий HAR + консоль + Azure App Insights + git blame для предклассификации каждого бага. Мобильное и реал-девайс тестирование через BrowserStack — критично для B2C-чекаута и BOPIS. Self-updating сьюты, где агенты наблюдают за падениями и code churn, открывают PR с предложениями апдейтов Steps/Assertions.

**Долгосрочно**: continuous production QA — теневой мониторинг реального трафика через App Insights для генерации synthetic regression-кейсов (естественное развитие сегодняшнего `/qa-monitoring`). ML test impact + cost optimizer, который выбирает самый дешёвый достаточный scope регресса под PR. Мульти-тенантное white-label тестирование.

Видение — полностью автономная QA-система, которая непрерывно тестирует, **сама чинит простые баги**, учится на падениях, обновляет собственные тест-кейсы на основе сигналов из продакшна и даёт real-time сигналы по качеству и стоимости — с минимальным ручным скриптингом.

Вот куда мы идём. Вопросы?
