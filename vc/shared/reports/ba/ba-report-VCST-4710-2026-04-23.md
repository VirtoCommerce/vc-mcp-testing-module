## Executive Summary

VCST-4710 introduces server-side **search, column sort, and faceted filtering (Country / State / City)** to the "Select Address" popup on the cart's Shipping Details section, spanning 4 open PRs across 4 repos. The shipped solution is server-side (new dedicated `currentCustomerAddresses` / `currentOrganizationAddresses` GraphQL queries with `keyword` + array facet args + `term_facets` response), not a client-side refinement of the legacy `contact.addresses` connection. PR #2219 is already deployed to QA and partially functional: the 3 facet dropdowns and keyword search are live, but **two P0 defects break core ticket ACs** — (1) column-header sort is declared in markup (`aria-sort="none"`) but not wired (no click handler, `cursor: auto`), and (2) the City facet shows postal codes instead of city names. Four lower-severity defects were also captured (wrong `aria-label` copy-pasted from BOPIS, mobile CITY filter clipped at 375px, Enter-to-search not supported, no pre-selection indicator). QA cannot fully validate the story until the two P0 defects are fixed and 20+ addresses are seeded in a test org.

## 3. API Analysis (Corrected Against Actual PR Implementation)

> The initial xAPI specialist analysis was performed against the live (pre-PR) `contact.addresses` / `organization.addresses` connections and concluded "no keyword, no facets — recommend Option A client-side." Once the 4 PRs were reviewed, that conclusion was **superseded** — the team shipped a **server-side** option with dedicated queries.

### Shipped GraphQL surface (PR #129)

```graphql
query GetCurrentUserAddresses(
  $after: String, $first: Int, $sort: SortInput,
  $countryCodes: [String!], $regionIds: [String!], $cities: [String!],
  $keyword: String
) {
  currentCustomerAddresses(
    after: $after, first: $first, sort: $sort,
    countryCodes: $countryCodes, regionIds: $regionIds, cities: $cities,
    keyword: $keyword
  ) {
    totalCount
    items { ...memberAddressFields }
    term_facets { name terms { term label count } }
  }
}
# currentOrganizationAddresses — identical arg signature, scoped to org
# checkDuplicateAddress(memberId, address) -> { isDuplicated: Boolean }
```

**memberAddressFields** (searchable by `keyword`): `name`, `line1`, `city`, `countryName`, `regionName`, `postalCode`.
**`term_facets` shape:** `[{ name, terms: [{ term, label, count }] }]`.
**Authorization:** `currentCustomerAddresses` scopes to `me`; `currentOrganizationAddresses` checks org membership → 401 no token, 403 wrong org.

### Frontend composable defaults (PR #2219)

- `useCustomerAddresses` — default sort `{ column: "lastName", direction: "Ascending" }`
- `useCurrentOrganizationAddresses` — default sort `{ column: "isFavorite", direction: "Descending" }`
- All filters + keyword sent server-side (no client-side filtering)
- `select-address-modal.vue` uses `paginationMode === "server"` branch

### Endpoint inventory

| Layer | Operation | Supports keyword? | Supports sort? | Supports facets? | Auth |
|---|---|---|---|---|---|
| GraphQL xAPI | `currentCustomerAddresses` | **Yes** | Yes (`SortInput`) | Yes (`term_facets` populated) | Storefront Bearer (me) |
| GraphQL xAPI | `currentOrganizationAddresses` | **Yes** | Yes | Yes | Storefront Bearer (org-scoped) |
| GraphQL xAPI | `checkDuplicateAddress` | N/A | N/A | N/A | Storefront Bearer |
| GraphQL xAPI (legacy) | `contact(id).addresses` | No | Yes | No (term_facets `[]`) | Storefront Bearer |
| REST | `POST /api/contacts/search` | Yes (member fields) | Yes | No | Platform Bearer |
| REST | `PUT /api/addresses?memberId` | N/A (write) | N/A | N/A | Platform Bearer |

### API health / recommendations

- **Legacy GraphQL fields (`contact.addresses`, `organization.addresses`) are now redundant** for the search/facet use case — the storefront should migrate remaining call sites to `currentCustomerAddresses` / `currentOrganizationAddresses` per PR #2219.
- **`term_facets` placeholder on the legacy connection** still exists but remains unpopulated. Consider deprecating or removing to avoid confusion.
- **Sort arg validation** — neither legacy nor new query validates allowed sort fields. Invalid keys should return an error, not silently fall back.
- **keyword search match fields** (name/line1/city/countryName/regionName/postalCode) should be documented explicitly; multi-word behavior and diacritic normalization need product confirmation.

## 5. User Documentation — Find your shipping address quickly

### Overview
The shipping address popup now includes search, sorting, and filters so you can locate the right address in seconds — even when your address book has dozens of entries.

### Opening the popup
1. Go to the **Cart** page.
2. In the **Shipping Details** section, click **"Please select a shipping address"**, or click the **pencil icon** next to an already-selected address.

### Searching by keyword
Type in the search bar; results filter across address name, street line, city, country, region, and postal code. Search is **case-insensitive**. Clear the text to restore the full list.

### Filtering by Country, State, and City
- Three facet filters at the top of the popup: **Country**, **State/Province**, **City**.
- Each option shows a **count badge**. Filters are **multi-select** and **combinable** across groups (e.g., Country = US + City = New York).
- Click a selected value again to deselect.

### Sorting
Click any **column header** to sort A→Z; click again for Z→A. Columns: **Address**, **Description**, **Country**, **Active**.
- Personal accounts default to last name ascending.
- Organization accounts show **favorites first** by default.

### Mobile
Addresses appear as stacked cards; filters are available in the same way as desktop.

### Adding a new address
Click **Add New** inside the popup to create an address without leaving checkout. Duplicate addresses are flagged for review before saving.

### Tips
- **Search before filtering** if you remember part of a street or city.
- **Combine filters** to zero in on a long list: Country → State → City.
- **Mark favorites** to keep frequent addresses at the top on org accounts.
- **Sort by Active** to spot deactivated entries.
- **Personal vs. org addresses** are separate contexts.
