### Risk Areas
- **PR build flag (XCart 3.1015.0-pr-119-6a09):** the above-stock-qty bug, conflicting subtotals, and silent-coupon issue all sit in cart math / mutation paths likely touched by PR #119. **Recommend not promoting this build until BL-CART-001 cap/reject behavior is restored.**
- The cart `selectedForCheckout` semantics surfaced again when adding a freshly-added line item â€” until the line-row checkbox is ticked, the line item is excluded from the Order Summary subtotal ($0). This is by-design per BL-CART-010ff, but the contrast between line-area "Subtotal: $170" and Order-Summary "Subtotal: $0" can mislead a B2B buyer mid-checkout. Worth a UX review of the line-row default-selection state on add.
- Mobile 375px qty stepper target is 32Ă—32 px â€” meets WCAG 2.5.8 AA (24Ă—24 min) but below 2.5.5 AAA recommendation (44Ă—44). Acceptable today; would fail an AAA audit.
- The "+ Add a gift" auto-promoted SHOT item appears intermittently with the regular cart â€” its lifecycle, and whether it should persist across Clear Cart, is not obvious to a fresh user.

### Observations
- **Rapid-fire qty stepper is debounced server-side.** 10 quick `+` clicks resulted in a single quantity increment (1 â†’ 2). No duplicate `addItem` mutations seen, no Apollo errors. Good throttling behavior.
- **Direct qty input of `0` removes the line item without confirmation.** Reasonable UX â€” but no undo, and no warning. Equivalent to a silent "remove from cart" via the qty field.
- **Decimal qty (`1.5`) is structurally impossible to enter** â€” the `<input type="number">` has no `step` attribute so the decimal point is stripped and `1` + `.` + `5` collapses to `15`. (Not strictly a bug; a behavior worth noting because a user trying to type "1" might accidentally fat-finger more digits with the dot key.)
- **BL-CART-003 coupon-on-sale math is correct.** Applying THRESH50 (10% off cart) to a $699 list / $255 post-sale subtotal yields exactly $25.50 in coupon savings â€” 10% of the post-sale figure, not 10% of the list. âś“
- **BL-CART-009 coupon swap is correct.** Switching from THRESH50 to FIXED5 cleanly removed THRESH50 and applied FIXED5 â€” discount delta dropped from $469.50 to $449.00, no double-stack.
- **BL-CART-007 same-product consolidation is correct.** Returning to the PDP after add shows the existing qty (`valuenow=4`), and further `+` clicks increment the existing line rather than creating a duplicate.
- **Multi-tab cart sync is live.** Mutating qty in tab B causes tab A to reflect the new state without manual refresh, suggesting WebSocket/SSE or aggressive cart polling.
- **Tax math is post-discount and stays at 20%** across all states tested ($699 â†’ $51, $932 â†’ $67). Matches BL-PRICE-002.

### Questions
- Is the cart's pre-existing "UNTUCKit eGift Card" (marked "no longer available for purchase") a stale fixture in TechFlow's account, or is this org's cart picking up residual lines from a deleted catalog product? If the latter, what happens on Place Order? Confirm with backend whether order creation rejects discontinued lines or silently drops them.
- Are coupon codes meant to be case-insensitive? If yes, lowercase entry should normalize to uppercase before submission. If no, the lowercase variant should surface the same "This code is not valid" feedback as `INVALIDCODE123`.
- The Order Summary subtotal showing **list Ă— qty** with a large "Discount" line that includes the catalog sale-vs-list delta is unusual for a B2B context â€” buyers will read this as "we're getting a $597 discount" when really $592 of that is the published sale price already in `placedPrice`. Is this the intended display contract, or should the Subtotal display the sale-price sum (matching the line-area "Subtotal")?
