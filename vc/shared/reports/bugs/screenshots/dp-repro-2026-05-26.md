## Anomalies discovered
- Calendar UI is **single-month** (Reka UI `vc-calendar--mode--single`), not dual-month as the task description suggested. Start and End each open their own independent single-month popover.
- End calendar opens at current month (May 2026), not anchored to the chosen Start month (March 2026). Minor UX friction — user must navigate the End calendar back independently. Not a defect of the filter contract being tested.
- Calendar day cells are `div[role="button"]` (not native `<button>`); they don't surface as accessibility-tree gridcell-buttons in the snapshot, so test automation needs `aria-label="Weekday, Month D, YYYY"` selectors. Worth noting for future date-picker tests.
- TZ offset on the lower bound: `2026-03-04` (UTC date local entry) was serialized as `2026-03-03T23:00:00.000Z` — looks like Europe/Helsinki+0200 client offset producing an inclusive lower bound of the local-day start. Filter end is `2026-03-08T22:59:59.999Z` — the local end-of-day. Consistent across both paths; not a bug, just env context.
