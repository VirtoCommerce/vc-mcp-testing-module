## Bugs Found

### BUG_082_001 — Impersonation Access Token Not Invalidated After Operator Reverts (VCST-4906 AC#6 violation) — **Critical · CONFIRMED**

- **Test case:** IMP-046 — "Previous Impersonation Token Invalidated After Revert (AC#6 Security)"
- **Business rules violated:** BL-AUTH-007, BL-AUTH-011
- **Edge case ref:** ECL-14.1
- **Severity:** Critical (security)
- **Confirmed:** **true** — independently reproduced by `qa-testing-expert` on `playwright-firefox` (cross-engine from preliminary chrome run), fresh session, 35.7s delta between revert and replay (well outside any race window)
- **Confirmation evidence:** `reports/regression/REG-2026-05-14-1543/imp-046-confirmation/` (HAR, JWT decode, pre/post-revert responses, network log, screenshots)

**Escalations discovered during confirmation:**

1. **xAPI GraphQL accepts the revoked token too** — `POST /graphql { me { ... } }` with the stale IMP_TOKEN returns David's full profile (HTTP 200, organizationId=6fb516c1 TechFlow). Two independent backend surfaces (Platform REST + xAPI GraphQL) accept the token — this is not a `/currentuser` caching artifact.
2. **Token survives operator sign-out** — IMP_TOKEN remains valid even after the operator (John) fully signs out of the storefront. **No backend path invalidates impersonation tokens at all** — they live their full 30-min `exp` regardless of revert OR sign-out.
3. **Zero revocation call attempted** — filter `revoke|revocation|sign-out|signout|logout|introspect` against the full session network log returned zero matches. The only auth call during revert was `POST /connect/token grant_type=impersonate&user_id=` (empty user_id signals revert) → 200, mints new operator token. Revert is a pure client-side token swap.
4. **JWT claims:** `iat` 14:52:09Z · `exp` 15:22:09Z · 30-minute lifetime · `sub`=David · `vc_operator_user_id`=John · `jti`=`f471f635-...`. Signature + `exp` are the only validity checks — no server-side session bookkeeping.

**Root cause hypothesis:** Frontend never calls a revoke endpoint and backend has no impersonation-session bookkeeping. `/connect/token grant_type=impersonate` mints a 30-min JWT that is server-trusted by signature + `exp` only; revert is a pure client-side token swap, leaving the impersonation JWT fully usable against Platform REST and xAPI GraphQL for its entire 30-min lifetime even after operator sign-out.

**Failed assertion:** `POST-REVERT: GET /api/platform/security/currentuser with captured impersonation token returns HTTP 401`

**Steps to reproduce:**
1. Sign in as John Mitchell (CanImpersonate permission, fixture: `test-john.mitchell-20260310@test-agent.com`).
2. Navigate to `/account/impersonate/ec3031ac-6dd9-42e9-b7a7-0c10d9aac07b` (David Kim).
3. Capture the impersonation `access_token` from `localStorage['auth'].access_token`.
4. Verify token valid: `GET {BACK_URL}/api/platform/security/currentuser` with `Bearer {token}` → HTTP 200, userName=test-david.kim.
5. Account menu → "Back to John Mitchell". Wait for revert (banner gone, session = John Mitchell).
6. Re-send the same GET with the **captured impersonation token**.

**Expected:** HTTP 401 — the impersonation `access_token` must be revoked server-side within the same request cycle as the revert (AC#6: "Previous Token MUST be invalidated").

**Actual:** HTTP 200 with `userName=test-david.kim-20260310@test-agent.com`. The token remains valid indefinitely after revert. An attacker who intercepts or logs the impersonation token can continue to act as the target user even after the operator has ended the session.

**Evidence:** `reports/regression/REG-2026-05-14-1543/imp-046-fail-token-not-invalidated.png`
