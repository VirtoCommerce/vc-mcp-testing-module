Env: vcst-qa @ Platform 3.1032.0 · Admin SPA · Edge · 2026-06-03
Path: Stores → B2B-store → Sitemaps widget

## New sitemap created
- **Location: `sitemap/seo.xml`** · Items location pattern: `{slug}` (default)
- Items added: **Catalog items** type ("Include product and categories") → drilled into store catalog **B2B-mixed**, Select All + Add selected → **20 category items** added (Books, Bolts, Consumer Electronics, SEED-20260518-Electronics, etc.); category items generate their product URLs at sitemap generation.
- Persisted immediately on "Add selected" (list grid shows `sitemap/seo.xml = 20`; Save button stayed disabled = no pending changes). Widget badge now **6**.
- NOTE: the form **rejects `sitemap.xml` as a location** — inline validation: *"Avoid using 'sitemap.xml' as the sitemap location"* / *"Specify a relative URL unique within the index sitemap, e.g. 'products.xml'"*. `sitemap.xml` is the auto-generated **index** that aggregates the individual files, so it must not be a manual entry. I used `sitemap/seo.xml` per the form's guidance.

## Generate / Download option
Sitemaps widget toolbar offers **"Download as ZIP"** and **"Export to store assets"** (the latter generates files into store assets, optionally by background job via the "Export sitemap files by job" setting). Not triggered — only recorded.

## Store SEO / robots settings observed
- Store Settings → **Store > SEO**: only one setting — **"SEO links" = Long**. No robots/noindex/search-engine toggle here.
- Store Settings → **Sitemap > General**: product priority 1.0, category 0.7, blog 0.5; product freq daily, category/blog weekly; "Include default store's language in URL", "Image inclusion", "Export sitemap files by job".
- **SEO widget** (store level) = SEO URL keyword/alias records only ("No data"); no robots control.
- **Finding:** there is NO admin store setting controlling `<meta name="robots" content="noindex">`. That noindex is a storefront frontend/theme/config concern, not configurable from the Store blade. The storefront `/sitemap.xml` 404 is also not fixable by adding a manual `sitemap.xml` entry (form blocks it); the index is generated by the platform/storefront, so the 404 likely points to a storefront routing/generation issue rather than missing admin config.

## Export / Generation (triggered 2026-06-03 ~10:49 AM)
- **"Export to store assets"** → confirm dialog "Base URL" pre-filled `https://vcst-qa-storefront.govirto.com` → Confirm.
- Ran as a background job with live progress ("Catalog: ...generated N of N records for category ...").
- **Job status: SUCCESS.** Start 10:49:29 → End 10:49:37 (~8s). Completion message: *"Sitemap export to store assets finished."*
- **Output location:** store assets — `https://vcst-qa.govirto.com/cms-content/assets/stores/B2B-store/sitemap/` (i.e. `cms-content/assets/stores/B2B-store/sitemap/`). The **index `sitemap.xml` IS auto-generated** there.
- Verified served index (`.../sitemap/sitemap.xml`) = valid `<sitemapindex>` listing: `seo.xml` (my new one, first), `b2b-store.xml`, `jewelry.xml`, `products.xml` — 4 of 6 (content.xml=2, catalog-b2b-mixed.xml=3 not emitted, likely empty/deduped). All `<loc>` use storefront base `vcst-qa-storefront.govirto.com/en-US/sitemap/...`.
- **"Download as ZIP"** also triggered → `sitemap-2026-06-03-7823fd28-...zip` downloaded (to test-results/edge/). Contains the same generated files incl. index `sitemap.xml`.
- **"Export sitemap files by job" setting = OFF** (Sitemap > Export to assets group) — yet the UI export still ran as a (foreground-tracked) job and completed.

### 404 root cause re-confirmed (the ticket's actual problem)
- Storefront serves the index at **`https://vcst-qa-storefront.govirto.com/sitemap/sitemap.xml` → 200 OK** (valid index).
- Storefront **root `https://vcst-qa-storefront.govirto.com/sitemap.xml` → still 404** (the path robots.txt references).
- So generation succeeded and files exist/are served; the 404 is a **storefront routing/path mismatch** (`/sitemap/sitemap.xml` works, `/sitemap.xml` does not) — NOT a missing admin sitemap config. Fix belongs in storefront routing, not the Stores module.
