## Слайд 1 — Заголовок

Привет всем. Сегодня я хочу показать вам то, что меняет подход к QA-инжинирингу.

Это наша **система агентного тестирования** — вместо того чтобы писать скрипты на Playwright, мы пишем обычные текстовые инструкции, и AI-агенты выполняют реальные тесты в реальных браузерах. 14 специализированных агентов, ~1 546 тест-кейсов, ни одной строчки кода.

Пять ключевых характеристик системы сразу видны на экране: 14 AI-агентов, 36 сьютов, 3 параллельных браузера, CI/CD автоматизация (в планах) и самовосстановление. Это не концепция — это то, что работает в продакшне прямо сейчас.

## Слайд 2 — Проблема и решение

Покажу проблему, которую мы решили. Слева — традиционная Playwright-автоматизация: требует инженерных навыков, ломается на каждом переименовании CSS-класса, высокая стоимость поддержки, не понимает бизнес-логику.

Справа — Agentic QA: промпты на естественном языке, самовосстановление через контекстное понимание, добавить сьют = добавить CSV-файл, агенты применяют BL-* инварианты к каждому тесту.

Посмотрите на сравнение кода внизу: 140 строк хрупкого скрипта против 3 строк на естественном языке, которые делают то же самое. Это не просто улучшение — это другая парадигма. Когда завтра `.submit-btn-v2` переименуют, традиционный тест сломается. Агентный тест просто найдёт кнопку по контексту — приоритет поиска: `data-testid` → ARIA-метки → видимый текст → расположение.

## Слайд 3 — Обзор проекта

Система тестирует B2B e-commerce платформу Virto Commerce — фронтенд, админку, REST API, GraphQL, оформление заказов, платежи, B2B мультиорганизации и доступность.

Три слова описывают её суть: **AI-Native** (тесты — это промпты, не код, LLM рассуждает о том, что нажать и что проверить), **Модульная** (36 CSV-сьютов, 14 агентов, 18 скилов — каждый можно улучшать независимо), **CI-интегрированная** (GitHub Actions запускает всё по расписанию с уведомлением в Teams — пока в планах).

Цифра, которую стоит запомнить: **~1 546 тест-кейсов** в 36 сьютах — 643 фронтенд-сценария и 903 бэкенд-сценария. Поддерживать это вручную — нереально. С нашей системой — практически нулевое обслуживание.

## Слайд 4 — Архитектура

Теперь, когда вы увидели что и зачем, покажу, как всё устроено. Три режима тестирования, одна общая основа.

**Интерактивный режим**: инженер вводит команду в IDE → Claude Code CLI → 14 агентов → 9 MCP-серверов → 3 реальных браузера. Лучше всего для спринтового тестирования, дебаггинга, тестирования конкретных тикетов.

**Agent Teams режим**: полностью автономный — `autonomous-orchestrator` создаёт команду через TeamCreate API, управляет токен-бакетом (3 слота браузера + 1 репортер), применяет экспоненциальный откат при сбоях, автоматически создаёт JIRA-тикеты для багов.

**CI-конвейер** (в планах): Docker-контейнер → Claude Agent SDK → Headless Chromium → уведомление в Teams. Запускается по расписанию GitHub Actions без участия человека.

Ключевой момент снизу: все три режима читают **одни и те же** `test-suites.json`, CSV-сьюты, определения агентов и файлы знаний. Одни тесты — три способа запуска.

Группы покрытия: **smoke** — 12 тестов перед деплоем, **critical** — ~120 P0-тестов, **sprint** — ~800 тестов, **full** — все ~1 546 кейсов. Выбираете нужный уровень одной командой.

## Слайд 6 — Команды агентов

Вот кто на самом деле выполняет работу. 14 агентов в четырёх слоях.

**Оркестраторы**: три агента координируют, планируют и маршрутизируют работу. `qa-lead-orchestrator` читает JIRA-тикеты, принимает решения GO/NO-GO. `regression-orchestrator` запускает параллельные сьюты с браузерным пулом. `autonomous-orchestrator` управляет режимом Agent Teams с токен-бакетом 3+1.

**Специалисты-исполнители**: четыре **Opus**-агента с глубоким браузерным рассуждением. Обратите внимание на браузеры — каждый агент получает свой изолированный браузер. `qa-frontend-expert` — Chrome, `qa-backend-expert` — Edge, `qa-testing-expert` — Firefox, `ui-ux-expert` — Chrome DevTools.

Внизу: **4 BA-агента** для бизнес-анализа без доступа к браузеру, и **test-management-specialist** для планов, RTM и матрицы покрытия.

## Слайд 7 — Команды

Десять команд — весь интерфейс для полного QA-цикла. Вводите команду в Terminal — система делает всё остальное.

**Верхний ряд**: `/qa-smoke` — 12 P0 тестов, два параллельных трека, вердикт GO/NO-GO за 15 минут. `/qa-test VCST-XXXX` — читает JIRA-тикет, запускает нужных специалистов, заводит баги с доказательной базой, переводит статус тикета. `/qa-regression sprint` — параллельный запуск 26 сьютов батчами по 3.

**Нижний ряд**: `/qa-bug` — воспроизводит, собирает HAR и скриншоты, заводит JIRA с P0–P3 severity. `/qa-status` и `/qa-env-check` — **автоматически вызываемые**, срабатывают при открытии проекта. У вас всегда актуальная картина здоровья среды без явного запроса.

## Слайд 8 — 9 MCP-серверов

MCP — Model Context Protocol — это то, что превращает Claude из генератора текста в агент тестирования. Без MCP он только описывает действия. С MCP — открывает реальные браузеры, кликает на реальные кнопки, сравнивает Figma-макеты, запрашивает Azure и экспортирует HAR-файлы.

Вверху на слайде показан полный поток: инженер вводит команду → Claude рассуждает и планирует → 9 MCP-серверов дают доступ к инструментам → 3 браузера параллельно → отчёты, скриншоты, HAR, JIRA.

Четыре категории внизу: **Browser Automation** — Chrome, Firefox, Edge плюс DevTools для трассировки. **Design Comparison** — Figma MCP для pixel-diff. **API & Code** — Postman + GitHub. **Cloud & Docs** — Azure App Insights + Context7 для документации.

*[~60 сек]*

## Слайд 9 — Бизнес-правила и граничные случаи

Два механизма, которые делают систему по-настоящему интеллектуальной.

**Бизнес-инварианты BL-*** (слева): все агенты разделяют `business-logic.md`. BL-PRICE-001: скидки не могут превышать 100%. BL-CART-004: товары без остатка нельзя купить. BL-AUTH-002: неаутентифицированные пользователи перенаправляются на логин. Когда агент обнаруживает нарушение, он не просто проваливает тест — он объясняет: «Нарушен BL-PRICE-001 — купон снизил цену ниже $0».

**ECL-* библиотека граничных случаев** (справа): 13 общих + 7 VC-специфичных паттернов с кросс-ссылками на BL-*. Каждый сгенерированный тест-кейс цитирует и BL-*, и ECL-*.

Внизу справа — поток: JIRA-тикет → автоматическое определение BL-* + ECL-* → тест-кейсы со ссылками на правила.

*[~60 сек]*

## Слайд 10 — Параллельная регрессия

Четыре ключевых слова: **smoke, critical, sprint, full**. Группы выбора под любой сценарий. Smoke — 1 сьют (~12 тестов) перед каждым деплоем. Critical — 4 P0-сьюта (~120 тестов) перед любым релизом. Sprint — 26 сьютов (~800 тестов) перед спринт-релизом. Full — все 36 (~1 546 тестов) перед продакшном.

Вверху слайда — визуализация браузерного пула: три полосы Chrome/Firefox/Edge выполняют сьюты одновременно. Оркестратор сортирует P0-сьюты первыми — если что-то критическое сломано, вы узнаёте сразу, не дожидаясь 30 других сьютов.

Качественные ворота (справа) автоматические:
- **APPROVED** — ≥95% pass rate, 0 P0-багов — безопасно деплоить
- **CONDITIONAL** — только P2/P3 — деплоить с задачами на исправление
- **BLOCKED** — P0/P1 сбой — **никакого деплоя, независимо от расписания**

При сбое — ретрай до 2 раз с fallback цепочкой: chrome → firefox → edge.

## Слайд 11 — CI/CD автоматизация (в планах)

Обратите внимание — этот конвейер в стадии планирования. Архитектура уже спроектирована.

GitHub Actions будет запускать дейли смоук с понедельника по пятницу в 6:00 UTC ($5 бюджет, Suite 01), полную регрессию каждое воскресенье в 2:00 UTC ($80 бюджет, все 36 сьютов). Ручной запуск доступен для любого объёма, любой среды через `workflow_dispatch`.

Слева — визуализация конвейера: Docker обеспечивает чистую воспроизводимую среду → Claude Agent SDK управляет оркестрацией → Headless Chromium выполняет тесты → отчёты в `reports/regression/ci-YYYY-MM-DD/` → Adaptive Card в Teams.

Справа — ключевые фичи: бюджетный контроль, 90-дневная история, 30-дневное хранение артефактов. Команда для запуска одной строкой: `docker run --env-file .env -e SUITE_SELECTION=critical vc-regression`.

## Слайд 12 — База знаний + скилы

Каждый агент умный отчасти благодаря общей базе знаний. Слева — **12 файлов**: бизнес-правила (BL-*), библиотека граничных случаев (ECL-*), паттерны платформы, карта сайта, пороги производительности. Обновите один файл — все 14 агентов немедленно узнают новое правило.

Справа — **18 скилов** в трёх категориях. `testing/` — 8 скилов для работы с браузерами и инструментами: Storybook, Accessibility, Design review, Test plans, Checklists, API testing, Coverage gap, Seed data. `qa-methodology/` — 9 скилов-процессов: ISTQB, SBTM, риск-матрица, метрики. `vc-knowledge/` — 1 скил `/vc-docs`, автоматически вызывается через Context7 при незнакомых концепциях.

Важный момент: это институциональные знания, которые не уходят вместе с сотрудником. Новый член команды? С первого дня у него доступ к той же методологии.

## Слайд 13 — Генерация тест-кейсов

Это одна из самых мощных фич: `/qa-test-cases-generator` создаёт тест-кейсы автоматически из JIRA-тикета, BDD-сценария, доменного чеклиста или устаревшего TestRail-сьюта.

Вверху — поток: JIRA-тикет → парсинг acceptance criteria + определение слоёв → генерация кейсов с BL-* и ECL-* ссылками → маршрутизация к нужному агенту.

Слева — пять входных форматов: `VCST-XXXX`, `domain`, `suite NN`, `from-checklist`, `from-bdd`. Справа — что включает каждый кейс: **BL-* ссылка** на бизнес-инвариант, **ECL-* паттерн** граничного случая, типизированные шаги (`[NAV]`, `[ACT]`, `[WAIT]`), теги ассертов (`[DOM]`, `[STATE]`, `[MATH]`) и как минимум 2 failure-сигнала. Генерируете один раз — кейсы готовы к исполнению агентом без изменений.

## Слайд 14 — Генерация тестовых данных

Перед запуском тестов нужны данные. `/qa-seed-data` создаёт полноценную тестовую среду через Postman MCP — каталоги, продукты, ценообразование, B2B-организации, пользователей — одной командой.

Вверху — поток: выбор профиля → Postman MCP строит и запускает коллекцию → тестовые данные с AGENT-TEST-* префиксом → запуск тестов → безопасный teardown.

Шесть профилей в таблице: **minimal** — 1 продукт для смоука, **catalog** — 3-уровневое дерево + 5 продуктов + мультивалюта, **b2b** — организация с 3 ролями, **pricing** — два прайс-листа с tiered prices, **full** — всё сразу, **teardown** — удаляет только AGENT-TEST-* сущности.

Правило безопасности: все сущности создаются с префиксом `AGENT-TEST-{дата}`. Teardown удаляет только сущности с таким именем — продакшн-данные никогда не затрагиваются.

## Слайд 15 — Сквозной поток /qa-test

Пройдёмся по тому, что реально происходит, когда вы вводите `/qa-test VCST-XXXX`.

Пять шагов на временной шкале:

**Шаг 1** — читаем JIRA-тикет через Atlassian MCP: заголовок, описание, acceptance criteria, определяем scope — витрина, админка или API.

**Шаг 2** — диспетчеризация специалистов: qa-frontend-expert (Chrome) берёт витринные AC, qa-backend-expert (Edge) — API и админку. Запускаются параллельно.

**Шаг 3** — выполнение: навигация к фиче, проверка каждого AC через реальный браузер, скриншоты + HAR + консоль.

**Шаг 4** — если что-то не прошло: баг-репорт с P0–P3 severity, скриншоты, HAR, ссылка на BL-* правило, автосоздание в JIRA.

**Шаг 5** — все AC прошли → тикет переводится в «Testing Complete» с привязкой доказательств. Баги найдены → блокировка релиза.

## Слайд 16 — Преимущества

Шесть сильных сторон, три из них особенно важны.

**Нулевое обслуживание** — UI-изменения не ломают тесты. Самовосстановление означает, что QA-команду не поднимают ночью каждый раз, когда рефакторят кнопку. Время на поддержку хрупких селекторов теперь идёт на написание нового покрытия.

**Понимание бизнеса** — агенты не просто обнаруживают сбои, они объясняют их в бизнес-терминах. «Нарушен BL-PRICE-001» говорит разработчику, куда смотреть. «AssertionError на строке 47» — не говорит ничего.

**Не нужен код** — QA-дисциплина открыта для всей команды. Продакт-менеджеры могут добавлять приёмочные тесты, дизайнеры — визуальные кейсы, джуниор QA — 50 тест-кейсов за один день. Нулевой порог входа.

Три цифры внизу говорят сами за себя: **~0** инцидентов с поддержкой селекторов за спринт, **15 минут** от команды до вердикта, **1 строка CSV** — стоимость добавления нового тест-кейса.

## Слайд 17 — Роадмап

Два пункта уже поставлены и используются ежедневно: **генератор AI тест-кейсов** (`/qa-test-cases-generator`) и **Agent Teams режим** — оба работают в продакшне прямо сейчас.

**Q2 2026** — три направления: мультибраузерный CI (расширяем с одного headless Chromium до Chrome + Firefox + Edge), визуальные регрессионные базлайны (персистентные снапшоты всех 55 Storybook-компонентов с pixel-diff против Figma), отчёты покрытия спринта (автоматический маппинг: какие PR-изменения затрагивают какие сьюты, авто-выбор scope для регрессии).

**Q3 2026**: мобильное тестирование через BrowserStack (iOS Safari iPhone 15 Pro, Android Chrome Samsung Galaxy S24 — критично для B2C checkout), самообновляющиеся сьюты (агент анализирует изменения кода, выявляет пробелы, предлагает новые тест-кейсы на review команде).

**Будущее**: регрессия производительности (Core Web Vitals baseline per release, LCP/CLS alerts), анализ влияния на тесты (ML-предсказание какие сьюты с наибольшей вероятностью упадут), мультитенантное тестирование.

**Долгосрочное видение**: полностью автономная QA-система. Непрерывное тестирование. Учится на ошибках. Обновляет свой тест-сьют. Ноль ручного написания скриптов.

Спасибо — вопросы?

## Слайд 18 — RAG + векторный поиск (планируется)

Последний слайд — это не что-то, что уже работает. Это следующий архитектурный шаг, который мы планируем реализовать.

Слева — проблема: сейчас каждый агент при запуске получает все 12 файлов знаний — около 100 КБ текста. Даже когда тестируется только авторизация, агент читает все правила ценообразования, все ECL-паттерны, всю карту сайта. Это расточительно и не масштабируется. Внизу — важная деталь: Context7 — это **уже** RAG. Скил `/vc-docs` делает семантический поиск по документации VC. Мы расширяем этот паттерн на всю базу знаний.

Справа — четыре конкретных значения:
- **Снижение стоимости токенов** — ~100 КБ контекст → ~5 КБ только нужного
- **Дедупликация тест-кейсов** — перед генерацией проверяем семантическое сходство с 1 546 существующими
- **Поиск похожих багов** — перед созданием JIRA проверяем эмбеддинги прошлых багов
- **Умный test impact analysis** — эмбеддинг git diff → семантическое сопоставление с тест-кейсами

Внизу — визуализация: Static Loading (~100 КБ всегда) vs RAG (~5 КБ релевантного). Разница в 20 раз.
