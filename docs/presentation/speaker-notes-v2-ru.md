## Слайд 1 — Заголовок

Привет всем. Сегодня я хочу показать вам то, что меняет подход к QA-инжинирингу.

Это наша **система агентного тестирования** — вместо того чтобы писать скрипты на Playwright, мы пишем обычные текстовые инструкции, и AI-агенты выполняют реальные тесты в реальных браузерах. 14 специализированных агентов, ~3 000 тест-кейсов, ни одной строчки кода.

Пять ключевых характеристик системы сразу видны на экране: 14 AI-агентов, 97 сьютов, 3 параллельных браузера, CI/CD автоматизация (в планах) и самовосстановление. Это не концепция — это то, что работает в продакшне прямо сейчас.

## Слайд 2 — Проблема и решение

Покажу проблему, которую мы решили. Слева — традиционная Playwright-автоматизация: требует инженерных навыков, ломается на каждом переименовании CSS-класса, высокая стоимость поддержки, не понимает бизнес-логику.

Справа — Agentic QA: промпты на естественном языке, самовосстановление через контекстное понимание, добавить сьют = добавить CSV-файл, агенты применяют BL-* инварианты к каждому тесту.

Посмотрите на сравнение кода внизу: 140 строк хрупкого скрипта против 3 строк на естественном языке, которые делают то же самое. Это не просто улучшение — это другая парадигма. Когда завтра `.submit-btn-v2` переименуют, традиционный тест сломается. Агентный тест просто найдёт кнопку по контексту — приоритет поиска: `data-testid` → ARIA-метки → видимый текст → расположение.

## Слайд 4 — Архитектура

Теперь, когда вы увидели что и зачем, покажу, как всё устроено. Три режима тестирования, одна общая основа.

**Интерактивный режим**: инженер вводит команду в IDE → Claude Code CLI → 14 агентов → 10 MCP-серверов → 3 реальных браузера. Лучше всего для спринтового тестирования, дебаггинга, тестирования конкретных тикетов.

**Agent Teams режим**: полностью автономный — `autonomous-orchestrator` создаёт команду через TeamCreate API, управляет токен-бакетом (3 слота браузера + 1 репортер), применяет экспоненциальный откат при сбоях, автоматически создаёт JIRA-тикеты для багов.

**CI-конвейер** (в планах): Docker-контейнер → Claude Agent SDK → Headless Chromium → уведомление в Teams. Запускается по расписанию GitHub Actions без участия человека.

Ключевой момент снизу: все три режима читают **одни и те же** `test-suites.json`, CSV-сьюты, определения агентов и файлы знаний. Одни тесты — три способа запуска.

## Слайд 6 — Команды агентов

Вот кто на самом деле выполняет работу. 14 агентов в четырёх слоях.

**Оркестраторы**: три агента координируют, планируют и маршрутизируют работу. `qa-lead-orchestrator` читает JIRA-тикеты, принимает решения GO/NO-GO. `regression-orchestrator` запускает параллельные сьюты с браузерным пулом. `autonomous-orchestrator` управляет режимом Agent Teams с токен-бакетом 3+1.

**Специалисты-исполнители**: четыре **Opus**-агента с глубоким браузерным рассуждением. Обратите внимание на браузеры — каждый агент получает свой изолированный браузер. `qa-frontend-expert` — Chrome, `qa-backend-expert` — Edge, `qa-testing-expert` — Firefox, `ui-ux-expert` — Chrome DevTools.

Внизу: **4 BA-агента** для бизнес-анализа без доступа к браузеру, и **test-management-specialist** для планов, RTM и матрицы покрытия.

## Слайд 7 — Команды

Шестнадцать команд — весь интерфейс для полного QA-цикла. Вводите команду в Terminal — система делает всё остальное.

**Верхний ряд**: `/qa-smoke` — 12 P0 тестов, два параллельных трека, вердикт GO/NO-GO за 15 минут. `/qa-test VCST-XXXX` — читает JIRA-тикет, запускает нужных специалистов, заводит баги с доказательной базой, переводит статус тикета. `/qa-regression sprint` — план-зависимый запуск (~88 сьютов, ~2 700 тестов) батчами по 3.

**Нижний ряд**: `/qa-bug` — воспроизводит, собирает HAR и скриншоты, заводит JIRA с P0–P3 severity. `/qa-status` и `/qa-env-check` — **автоматически вызываемые**, срабатывают при открытии проекта. У вас всегда актуальная картина здоровья среды без явного запроса.

## Слайд 8 — Pipeline и Quality команды

Три команды, которые завершают полный QA-цикл — следующий слайд выделяет им отдельный экран, потому что они новые и каждая делает то, чего нет в первой шестёрке.

**`/qa-test-lifecycle`** — единый pipeline-конвейер. Передаёте ему ID сьюта, домен, JIRA-тикет, номер PR, имя модуля или сырой git diff — он определяет область, синхронизирует устаревшие Steps/Assertions, анализирует пробелы покрытия, генерирует недостающие кейсы, прогоняет 7-мерное качественное ревью, исправляет что может, верифицирует результат и только потом одобряет. Именно это удерживает ~3 000-кейсовый сьют от деградации, пока код меняется под ним.

**`/qa-design`** делает двойной аудит — сначала в Storybook в изоляции, потом на живой витрине в интеграции. Это разделение ловит два разных класса багов, которые однослойные аудиты пропускают: баги, проявляющиеся только в изолированном рендере компонента (неправильные дефолтные пропы, отсутствующие slot-фоллбэки), и баги, проявляющиеся только при подключении к реальной странице (коллизии состояний, layout-сдвиги под реальными данными). Область задаётся matrix-driven по critical-UI scope, с эвристическим фоллбэком для off-matrix целей.

**`/qa-verify-fix VCST-XXXX`** замыкает JIRA-петлю. Подтягивает тикет, воспроизводит исходные STR (шаги воспроизведения), подтверждает что фикс действительно работает, прогоняет релевантные регрессионные проверки чтобы убедиться что ничего больше не сломалось, и переводит тикет. Используется после того, как разработчик отметил фикс готовым — повторное ручное выполнение не требуется.

Вместе эти три команды превращают QA-систему из «запускает тесты по запросу» в «автоматически держит весь тестовый актив и баг-пайплайн в синхроне с кодовой базой».

## Слайд 9 — 10 MCP-серверов

MCP — Model Context Protocol — это то, что превращает Claude из генератора текста в агент тестирования. Без MCP он только описывает действия. С MCP — открывает реальные браузеры, кликает на реальные кнопки, сравнивает Figma-макеты, запрашивает Azure и экспортирует HAR-файлы.

Вверху на слайде показан полный поток: инженер вводит команду → Claude рассуждает и планирует → 10 MCP-серверов дают доступ к инструментам → 3 браузера параллельно → отчёты, скриншоты, HAR, JIRA.

Четыре категории внизу: **Browser Automation** — Chrome, Firefox, Edge плюс DevTools для трассировки. **Design Comparison** — Figma MCP для pixel-diff. **API & Code** — Postman + GitHub. **Cloud & Docs** — Azure App Insights + Context7 для документации.

## Слайд 10 — Бизнес-правила и граничные случаи

Два механизма, которые делают систему по-настоящему интеллектуальной.

**Бизнес-инварианты BL-*** (слева): все агенты разделяют `business-logic.md`. BL-PRICE-001: скидки не могут превышать 100%. BL-CART-004: товары без остатка нельзя купить. BL-AUTH-002: неаутентифицированные пользователи перенаправляются на логин. Когда агент обнаруживает нарушение, он не просто проваливает тест — он объясняет: «Нарушен BL-PRICE-001 — купон снизил цену ниже $0».

**ECL-* библиотека граничных случаев** (справа): 13 общих + 7 VC-специфичных паттернов с кросс-ссылками на BL-*. Каждый сгенерированный тест-кейс цитирует и BL-*, и ECL-*.

Внизу справа — поток: JIRA-тикет → автоматическое определение BL-* + ECL-* → тест-кейсы со ссылками на правила.

*[~60 сек]*

## Слайд 11 — Параллельная регрессия

Четыре ключевых слова: **smoke, critical, sprint, full**. Группы выбора под любой сценарий. Smoke — 2 сьюта (~140 тестов) перед каждым деплоем. Critical — 5 P0-сьютов (~230 тестов) перед любым релизом. Sprint — план-зависимо (~88 сьютов, ~2 700 тестов) перед спринт-релизом. Full — все 97 (~3 000 тестов) перед продакшном.

Вверху слайда — визуализация браузерного пула: три полосы Chrome/Firefox/Edge выполняют сьюты одновременно. Оркестратор сортирует P0-сьюты первыми — если что-то критическое сломано, вы узнаёте сразу, не дожидаясь 30 других сьютов.

Качественные ворота (справа) автоматические:
- **APPROVED** — ≥95% pass rate, 0 P0-багов — безопасно деплоить
- **CONDITIONAL** — только P2/P3 — деплоить с задачами на исправление
- **BLOCKED** — P0/P1 сбой — **никакого деплоя, независимо от расписания**

При сбое — ретрай до 2 раз с fallback цепочкой: chrome → firefox → edge.

## Слайд 14 — GraphQL xAPI: схема + runner-native тесты

Два knowledge-файла плюс канонический раннер превращают xAPI GraphQL-поверхность в детерминированные, валидируемые по схеме CSV-тест-кейсы — без единого кастомного JS-скрипта.

Слева — два knowledge-файла. **`graphql-schema.md`** — снимок живой интроспекции схемы xAPI: queries, mutations, input/return типы. Каждый агент, который пишет или ревьюит GraphQL-запрос, **сначала сверяется со схемой** — это единственный надёжный способ не выдумать несуществующее поле. **`graphql-test-cases-runner.md`** — канонический контракт авторинга runner-native CSV-кейсов: полная грамматика тегов, формы предикатов, цепочки capture, типичные ошибки, чеклист авторинга и worked example.

Справа — грамматика тегов, которую интерпретирует раннер. Шаги: `[AUTH]`, `[GQL-OP]`, `[GQL-VARS]`, `[GQL-EXEC]`, `[GQL-CAPTURE]`, `[REST-OP]`, `[REST-EXEC]`, `[REST-CAPTURE]`. Ассерты: `[ERRORS]`, `[DATA]`, `[NULL]`, `[COUNT]`, `[VAR]`. Фильтры: getByPath-синтаксис, арифметика, cross-path, OR-AND предикаты.

Внизу — сам раннер: **`scripts/graphql-runner.ts`**. Он делает валидацию по схеме, подстановку переменных, резолв `@td()`, захват доказательств. **Никогда не пишите кастомные JS-скрипты для исполнения GraphQL CSV-кейсов** — всегда используйте канонический раннер. Команда: `npx tsx scripts/graphql-runner.ts --case <csv>:<ID>`. Эталонный сьют для подражания: `050i-graphql-configurations.csv`.

## Слайд 15 — Live Test-Data Discovery — без хардкода ID

Каталоги пересевают. B2B-организации пересоздают. Корни виртуальных каталогов мигрируют. Захардкоженные ID и SKU гниют быстро. Live discovery резолвит сущности в рантайме против актуальной среды.

Слева — knowledge-файл **`live-discovery.md`**: дерево решений и рецепты — когда выбирать `{{VAR}}`, когда `@td()`, когда `live-discover`, когда `random-data`. Покрывает изоляцию параллельных прогонов против пула агент-юзеров и конвенции очистки `AGENT-TEST-*`.

Четыре слоя тестовых данных в таблице ниже:
- **`{{VAR}}`** — из `.env` (33 переменные). Для URL, кредов, контекста среды.
- **`@td()`** — из `aliases.json` + CSV. Для именованных сущностей, на которые мы ассертим по имени (конкретный configurable laptop, известный купон).
- **`live-discover`** — xAPI в рантайме. Для «первый доступный продукт» — ассертим форму, не значение.
- **`random-data`** — zero-dep генераторы. Для уникальных входных данных, на которые мы не ассертим точные значения (email-регистрации, имена организаций, BVA-количества).

Справа — два скрипта-помощника. **`scripts/lib/live-discover.ts`** — типизированные xAPI-примитивы: корень каталога, продукты по фильтру, адреса, состояние корзины, активные купоны. Возвращает реальную сущность из текущей среды, никогда не захардкоженный ID. **`scripts/lib/random-data.ts`** — zero-dep генераторы: email, имена организаций, SKU, количества, комментарии. По умолчанию префикс `AGENT-TEST-{date}` — `/qa-seed-data teardown` подметает их безопасно.

Антипаттерн: захардкодить GUID продукта или SKU прямо в колонке `Test_Data`. Литерал, который не `{{VAR}}` и не `@td()`, — это автоматический фейл ревью (Dimension 6 — Data Validity).

## Слайд 16 — Теги Test Runner + резолвер `@td()`

Два knowledge-файла плюс резолвер и валидатор превращают CSV-строки в детерминированные, исполняемые тест-кейсы — интерпретируемые одинаково и стандартным, и autonomous раннером.

Слева — knowledge-файл **`test-runner-tags.md`**: общий справочник по CSV-колонкам, тегам шагов, ассертов и cross-layer-тегов для браузерных тестов. Используется и `test-runner-agent`, и `autonomous-test-runner` — одна и та же грамматика, два разных раннера.

Категории тегов:
- **Step tags**: `[NAV]`, `[ACT]`, `[WAIT]`, `[LOGIN]`, `[SETUP]`
- **Assertion tags**: `[DOM]`, `[STATE]`, `[MATH]`, `[API]`, `[CONSOLE]`
- **Cross-layer**: `[HAR]`, `[SCREEN]`, `[PERF]`

Справа — резолвер `@td(ALIAS.field)`. Тестовые данные резолвятся в рантайме, никогда не хардкодятся. Резолвер читает `test-data/aliases.json` → строки CSV в `test-data/`. Каталоги пересевают, ID меняются — alias остаётся стабильным; одно обновление CSV распространяется на всех потребителей.

Реализация: `scripts/lib/test-data-resolver.ts` (резолвер), `test-data/aliases.json` (реестр с `_meta.version`), `test-data/*.csv` (строки сущностей).

Валидация: каждая ссылка `@td()` проверяется до прогона. Команда: `npx tsx scripts/validate-td-refs.ts`. Кроме того, `graphql-runner.ts` отвергает нерезолвленные токены ещё на lint-этапе.

## Слайд 17 — Critical UI Scope — регрессионный чек-лист компонентов

Матрица 7×8, которая фиксирует, какие BL-UI инварианты применимы к какому компоненту на какой странице. Валидатор и хелпер измерения layout удерживают матрицу от тихого дрейфа.

Слева — knowledge-файл **`critical-ui-scope.md`**: регрессионный чек-лист **7 компонентов × 8 страниц** с применимыми BL-UI-* инвариантами в каждой клетке.

7 компонентов: `VcButton`, `VcProductCard`, `VcLineItem`, `VcTable`, `VcDialog`, `Popover`, `VcSidebar`.
8 страниц: `/`, `/catalog`, PDP, `/cart`, `/account/orders`, `/account/lists`, `/company/members`, `/company/info`.

Инварианты BL-UI-001..010: видимость focus-ring, hover-состояние, disabled-состояние, loading-skeleton, стабильность layout под данными, empty-state, error-state, keyboard-навигация.

Справа — измерение и enforcement. **`scripts/lib/measure-layout.ts`** — хелпер измерения layout, используемый BL-UI тестами: измеряет размеры, дельты позиций, CLS-вклады по каждому компоненту.

Покрывающий сьют: **`048b-layout-stability.csv`**, селекция `layout-stability`.

Enforcement: **`npm run scope:validate`** выходит с ненулевым кодом, если любая покрытая клетка матрицы указывает на отсутствующий test ID. Это превращает чек-лист из декларации в CI-проверку.

## Слайд 18 — Генерация тест-кейсов

Это одна из самых мощных фич: `/qa-test-cases-generator` создаёт тест-кейсы автоматически из JIRA-тикета, BDD-сценария, доменного чеклиста или устаревшего TestRail-сьюта.

Вверху — поток: JIRA-тикет → парсинг acceptance criteria + определение слоёв → генерация кейсов с BL-* и ECL-* ссылками → маршрутизация к нужному агенту.

Слева — пять входных форматов: `VCST-XXXX`, `domain`, `suite NN`, `from-checklist`, `from-bdd`. Справа — что включает каждый кейс: **BL-* ссылка** на бизнес-инвариант, **ECL-* паттерн** граничного случая, типизированные шаги (`[NAV]`, `[ACT]`, `[WAIT]`), теги ассертов (`[DOM]`, `[STATE]`, `[MATH]`) и как минимум 2 failure-сигнала. Генерируете один раз — кейсы готовы к исполнению агентом без изменений.

*[~60 сек]*

## Слайд 19 — Генерация тестовых данных

Перед запуском тестов нужны данные. `/qa-seed-data` создаёт полноценную тестовую среду через Postman MCP — каталоги, продукты, ценообразование, B2B-организации, пользователей — одной командой.

Вверху — поток: выбор профиля → Postman MCP строит и запускает коллекцию → тестовые данные с AGENT-TEST-* префиксом → запуск тестов → безопасный teardown.

Шесть профилей в таблице: **minimal** — 1 продукт для смоука, **catalog** — 3-уровневое дерево + 5 продуктов + мультивалюта, **b2b** — организация с 3 ролями, **pricing** — два прайс-листа с tiered prices, **full** — всё сразу, **teardown** — удаляет только AGENT-TEST-* сущности.

Правило безопасности: все сущности создаются с префиксом `AGENT-TEST-{дата}`. Teardown удаляет только сущности с таким именем — продакшн-данные никогда не затрагиваются.

*[~55 сек]*

## Слайд 20 — Сквозной поток /qa-test

Пройдёмся по тому, что реально происходит, когда вы вводите `/qa-test VCST-XXXX`.

Пять шагов на временной шкале:

**Шаг 1** — читаем JIRA-тикет через Atlassian MCP: заголовок, описание, acceptance criteria, определяем scope — витрина, админка или API.

**Шаг 2** — диспетчеризация специалистов: qa-frontend-expert (Chrome) берёт витринные AC, qa-backend-expert (Edge) — API и админку. Запускаются параллельно.

**Шаг 3** — выполнение: навигация к фиче, проверка каждого AC через реальный браузер, скриншоты + HAR + консоль.

**Шаг 4** — если что-то не прошло: баг-репорт с P0–P3 severity, скриншоты, HAR, ссылка на BL-* правило, автосоздание в JIRA.

**Шаг 5** — все AC прошли → тикет переводится в «Testing Complete» с привязкой доказательств. Баги найдены → блокировка релиза.

*[~65 сек]*

## Слайд 21 — Преимущества

Шесть сильных сторон, три из них особенно важны.

**Нулевое обслуживание** — UI-изменения не ломают тесты. Самовосстановление означает, что QA-команду не поднимают ночью каждый раз, когда рефакторят кнопку. Время на поддержку хрупких селекторов теперь идёт на написание нового покрытия.

**Понимание бизнеса** — агенты не просто обнаруживают сбои, они объясняют их в бизнес-терминах. «Нарушен BL-PRICE-001» говорит разработчику, куда смотреть. «AssertionError на строке 47» — не говорит ничего.

**Не нужен код** — QA-дисциплина открыта для всей команды. Продакт-менеджеры могут добавлять приёмочные тесты, дизайнеры — визуальные кейсы, джуниор QA — 50 тест-кейсов за один день. Нулевой порог входа.

Три цифры внизу говорят сами за себя: **~0** инцидентов с поддержкой селекторов за спринт, **15 минут** от команды до вердикта, **1 строка CSV** — стоимость добавления нового тест-кейса.

## Слайд 22 — Роадмап

Два пункта уже поставлены и используются ежедневно: **генератор AI тест-кейсов** (`/qa-test-cases-generator`) и **Agent Teams режим** — оба работают в продакшне прямо сейчас.

**Q2 2026** — три направления: мультибраузерный CI (расширяем с одного headless Chromium до Chrome + Firefox + Edge), визуальные регрессионные базлайны (персистентные снапшоты всех 55 Storybook-компонентов с pixel-diff против Figma), отчёты покрытия спринта (автоматический маппинг: какие PR-изменения затрагивают какие сьюты, авто-выбор scope для регрессии).

**Q3 2026**: мобильное тестирование через BrowserStack (iOS Safari iPhone 15 Pro, Android Chrome Samsung Galaxy S24 — критично для B2C checkout), самообновляющиеся сьюты (агент анализирует изменения кода, выявляет пробелы, предлагает новые тест-кейсы на review команде).

**Будущее**: регрессия производительности (Core Web Vitals baseline per release, LCP/CLS alerts), анализ влияния на тесты (ML-предсказание какие сьюты с наибольшей вероятностью упадут), мультитенантное тестирование.

**Долгосрочное видение**: полностью автономная QA-система. Непрерывное тестирование. Учится на ошибках. Обновляет свой тест-сьют. Ноль ручного написания скриптов.

Спасибо — вопросы?

## Советы по проведению

**Для 10-минутного lightning talk**: слайды 1, 2, 5, 6, 8, 11, 21, 22. Пропустите слайды 12–20.

**Для технической аудитории**: уделите больше времени слайдам 8 (lifecycle-команды), 10 (BL-* + ECL-* правила), **14–17** (GraphQL раннер, live discovery, теги, critical UI scope — это самые «инженерные» слайды) и 4 (три режима). Это то, что вызывает «вау» у разработчиков.

**Для менеджмента**: акцент на слайдах 5 (цифры), 21 (преимущества + три KPI) и 22 (роадмап). Опустите слайды 10, 13–19.

**Перед живой демо**: проверьте, что `.env` настроен, Chrome закрыт (конфликт user data dir), и введите `/qa-env-check` перед презентацией.
