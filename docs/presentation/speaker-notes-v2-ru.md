## Слайд 2 — Обзор проекта

Система тестирует B2B e-commerce платформу Virto Commerce — фронтенд, админку, REST API, GraphQL, оформление заказов, платежи, B2B мультиорганизации и доступность.

Цифра, которую стоит запомнить: **~1 546 тест-кейсов** в 36 сьютах — 643 фронтенд-сценария и 903 бэкенд-сценария. Поддерживать это вручную — нереально. С нашей системой — практически нулевое обслуживание.

## Слайд 3 — Проблема и решение

Покажу проблему, которую мы решили. Слева — традиционная Playwright-автоматизация: требует инженерных навыков, ломается на каждом переименовании CSS-класса, высокая стоимость поддержки, не понимает бизнес-логику.

Справа — Agentic QA: промпты на естественном языке, самовосстановление через контекстное понимание, добавить сьют = добавить CSV-файл, агенты применяют BL-* инварианты к каждому тесту.

Посмотрите на сравнение кода: 140 строк хрупкого скрипта против 3 строк на естественном языке, которые делают то же самое. Это не просто улучшение — это другая парадигма. Когда завтра `.submit-btn-v2` переименуют, традиционный тест сломается. Агентный тест просто найдёт кнопку по контексту.

## Слайд 5 — Команды агентов

Вот кто на самом деле выполняет работу. 14 агентов в четырёх слоях.

**Слой 1 — Оркестрация**: три Sonnet-агента координируют, планируют и маршрутизируют работу. `qa-lead-orchestrator` читает JIRA-тикеты, принимает решения GO/NO-GO. `regression-orchestrator` запускает параллельные сьюты с браузерным пулом. `autonomous-orchestrator` управляет режимом Agent Teams с токен-бакетом 3+1.

**Слой 2 — Специалисты-исполнители**: четыре **Opus**-агента с глубоким браузерным рассуждением. Обратите внимание на браузеры — каждый агент получает свой изолированный браузер. `qa-frontend-expert` — Chrome, `qa-backend-expert` — Edge, `qa-testing-expert` — Firefox, `ui-ux-expert` — Chrome DevTools.

**Слои 3 и 4**: планирование, тест-менеджмент и бизнес-анализ. `test-runner-agent` — параметризованный шаблон, создаётся на каждый сьют. BA-команда читает код и создаёт бизнес-артефакты без доступа к браузеру.

## Слайд 8 — 9 MCP-серверов

MCP — Model Context Protocol — это то, что превращает Claude из генератора текста в агент тестирования. Без MCP он только описывает действия. С MCP — открывает реальные браузеры, кликает на реальные кнопки, сравнивает Figma-макеты, запрашивает Azure и экспортирует HAR-файлы.

**Браузерная автоматизация** (4): Playwright Chrome, Firefox, Edge — три изолированных контекста для параллельного тестирования. Chrome DevTools MCP — консоль, сеть, трассировка производительности, экспорт HAR.

**Dev & Design интеграции** (3): Figma для сравнения с макетами, GitHub для ревью PR и поиска кода, Postman для API-тестирования и создания тестовых данных.

**Знания и облако** (2): Context7 для актуальной документации Virto Commerce, Azure MCP для App Insights логов и запросов к облачным ресурсам.

Практический эффект справа: агенты могут делать всё это в одном прогоне, автоматически — навигация в браузере, скриншот, диффинг с Figma, запрос Azure Insights, запуск Postman-коллекции.

## Слайд 9 — Самовосстановление + бизнес-правила

Два механизма, которые делают систему по-настоящему интеллектуальной.

**Самовосстановление**: агенты находят элементы по контексту, а не по селекторам. Приоритет: `data-testid` → ARIA-метки → видимый текст → расположение. Когда завтра разработчик переименует CSS-класс — что сломало бы 50 традиционных тестов — агенты этого не замечают. Цитата на слайде говорит всё: «Найди кнопку "Добавить в корзину" рядом с ценой продукта и нажми её» — работает независимо от CSS.

**Бизнес-инварианты BL-***: все агенты разделяют `business-logic.md`. BL-PRICE-001: скидки не могут превышать 100%. BL-CART-004: товары без остатка нельзя купить. BL-AUTH-002: неаутентифицированные пользователи перенаправляются на логин. Когда агент обнаруживает нарушение, он не просто проваливает тест — он объясняет: «Нарушен BL-PRICE-001 — купон снизил цену ниже $0».

**ECL-* библиотека граничных случаев**: 13 общих + 7 VC-специфичных паттернов. Каждый сгенерированный тест-кейс цитирует и BL-*, и ECL-*. Ни один граничный случай не выпадает.

## Слайд 10 — Генерация тест-кейсов

Это одна из самых мощных фич: `/qa-test-cases-generator` создаёт тест-кейсы автоматически из JIRA-тикета, BDD-сценария, доменного чеклиста или устаревшего TestRail-сьюта.

Запускаете `/qa-test-cases-generator VCST-4565` — скил получает тикет, **автоматически определяет затронутые слои** (новый REST endpoint → api + admin + e2e; GraphQL → graphql + e2e; фича витрины → storefront + graphql + e2e; кросс-доменная фича вроде купонов → все слои) и генерирует кейсы для каждого слоя.

Формат — ключевой. **15-колоночный обогащённый CSV**: каждый кейс содержит бизнес-правило (`BL-*`), ссылки на граничные случаи (`ECL-*`), типизированные шаги (`[NAV]`, `[ACT]`, `[WAIT]`), теги ассертов (`[DOM]`, `[STATE]`, `[MATH]`) и кросс-слоевые проверки (`[API]`, `[CONSOLE]`, `[NETWORK]`).

Таблица маршрутизации автоматически направляет каждый слой к нужному агенту и браузеру. Генерируете один раз — кейсы готовы к исполнению агентом без изменений.

*[~65 сек]*

## Слайд 11 — Генерация тестовых данных

Перед запуском тестов нужны данные. `/qa-seed-data` создаёт полноценную тестовую среду через Postman MCP — каталоги, продукты, ценообразование, B2B-организации, пользователей — одной командой.

Шесть профилей под любую задачу: **minimal** — 1 продукт, 1 цена, для быстрого смоука; **catalog** — 3-уровневое дерево категорий, 5 продуктов (физические/цифровые/конфигурируемые), мультивалютные цены; **b2b** — полная организация с 3 ролями (admin/buyer/viewer); **full** — всё сразу перед крупной регрессией.

Рабочий процесс (показан справа): проверка существующих коллекций → создание окружения из `.env` → построение seed-коллекции в правильном порядке зависимостей → запуск через `runCollection` → верификация через REST + GraphQL + поисковый индекс.

Правило безопасности: все сущности создаются с префиксом `AGENT-TEST-{дата}`. Teardown удаляет только сущности с таким именем — продакшн-данные никогда не затрагиваются, даже если teardown запустили не в той среде.

## Слайд 12 — Архитектура

Теперь, когда вы видели все части, покажу, как они складываются вместе. Три режима тестирования, одна общая основа.

**Интерактивный режим**: инженер вводит команду в IDE → Claude Code CLI → 14 агентов → 9 MCP-серверов → 3 реальных браузера. Лучше всего для спринтового тестирования, дебаггинга, тестирования конкретных тикетов.

**Agent Teams режим**: полностью автономный — `autonomous-orchestrator` создаёт команду через TeamCreate API, управляет токен-бакетом (3 слота браузера + 1 репортер), применяет экспоненциальный откат при сбоях, автоматически создаёт JIRA-тикеты для багов.

**CI-конвейер**: Docker-контейнер → Claude Agent SDK → Headless Chromium → уведомление в Teams. Запускается по расписанию GitHub Actions без участия человека.

Ключевой момент снизу: все три режима читают **одни и те же** `test-suites.json`, CSV-сьюты, определения агентов и файлы знаний. Одни тесты — три способа запуска.

## Слайд 13 — Параллельная регрессия

Четыре ключевых слова: **smoke, critical, sprint, full**. Группы выбора под любой сценарий. Smoke — 1 сьют (~12 тестов) перед каждым деплоем. Critical — 4 P0-сьюта (~120 тестов) перед любым релизом. Sprint — 26 сьютов (~800 тестов) перед спринт-релизом. Full — все 36 (~1 546 тестов) перед продакшном.

Оркестратор сортирует P0-сьюты первыми — если что-то критическое сломано, вы узнаёте сразу, не дожидаясь 30 других сьютов. Пакетирует по 3 под браузерный пул. При сбое — ретрай до 2 раз с fallback цепочкой: chrome → firefox → edge.

Качественные ворота автоматические:
- **APPROVED** — ≥95% pass rate, 0 P0-багов — безопасно деплоить
- **CONDITIONAL** — только P2/P3 — деплоить с задачами на исправление
- **BLOCKED** — P0/P1 сбой — **никакого деплоя, независимо от расписания**

## Слайд 14 — CI/CD автоматизация

Конвейер полностью автоматизирован. GitHub Actions запускает дейли смоук с понедельника по пятницу в 6:00 UTC ($5 бюджет, Suite 01), полную регрессию каждое воскресенье в 2:00 UTC ($80 бюджет, все 36 сьютов). Ручной запуск доступен для любого объёма, любой среды через `workflow_dispatch`.

Внутри конвейера: Docker обеспечивает чистую воспроизводимую среду. Claude Agent SDK управляет оркестрацией с теми же правилами, что и интерактивный режим. Бюджет отслеживается глобально — минимум $2 на сьют, оркестратор останавливает запуск новых сьютов при исчерпании.

Результаты — `reports/regression/ci-YYYY-MM-DD/` с markdown + JSON. Adaptive Card Teams попадает в QA-канал с процентами pass/fail, найденными багами и ссылкой на полный отчёт.

Команда для запуска одной строкой (справа на слайде): `docker run --env-file .env -e SUITE_SELECTION=critical -e TEST_ENVIRONMENT=qa vc-regression`.

## Слайд 15 — Workflow

Пройдёмся по тому, что реально происходит, когда вы вводите `/qa-smoke`.

**Слева — поток смоука** (6 шагов): вводите команду → оркестратор читает Suite 01, генерирует run ID `SMOKE-2026-03-12-0900` → запускает два параллельных агента → **Track A** (qa-frontend-expert, Chrome): 12 смоук-тестов витрины — вход, каталог, корзина, купон, чекаут → **Track B** (qa-backend-expert, Edge): здоровье админки — модули активны, API отвечают, GraphQL работает → оркестратор собирает результаты и выносит вердикт: checkout упал → NO-GO, все 12 прошли → GO → записывает `smoke-report.md`.

**Справа — поток расследования бага** (5 фаз): Воспроизвести → Изолировать (Chrome DevTools: консоль + HAR + сеть) → Корневая причина (сопоставить с BL-* и ECL-*) → Пакет доказательств (скриншот + HAR + консоль + шаги воспроизведения) → Завести JIRA с P0–P3 severity и всеми доказательствами.

Всё это — **15 минут** от команды до вердикта.

## Слайд 16 — Преимущества

Шесть сильных сторон, три из них особенно важны.

**Нулевое обслуживание** — UI-изменения не ломают тесты. Самовосстановление означает, что QA-команду не поднимают ночью каждый раз, когда рефакторят кнопку. Время на поддержку хрупких селекторов теперь идёт на написание нового покрытия.

**Понимание бизнеса** — агенты не просто обнаруживают сбои, они объясняют их в бизнес-терминах. «Нарушен BL-PRICE-001» говорит разработчику, куда смотреть. «AssertionError на строке 47» — не говорит ничего.

**Не нужен код** — QA-дисциплина открыта для всей команды. Продакт-менеджеры могут добавлять приёмочные тесты, дизайнеры — визуальные кейсы, джуниор QA — 50 тест-кейсов за один день. Нулевой порог входа.

Три цифры внизу говорят сами за себя: **~0** инцидентов с поддержкой селекторов за спринт, **15 минут** от команды до вердикта, **1 строка CSV** — стоимость добавления нового тест-кейса.

## Слайд 18 — RAG + векторный поиск (планируется)

Последний слайд — это не что-то, что уже работает. Это следующий архитектурный шаг, который мы планируем реализовать.

Сейчас каждый агент при запуске получает все 12 файлов знаний — около 100 КБ текста. Даже когда тестируется только авторизация, агент читает все правила ценообразования, все ECL-паттерны, всю карту сайта. Это расточительно и не масштабируется по мере роста базы знаний.

RAG решает это. Все знания — BL-* правила, ECL-* паттерны, 1 546 тест-кейсов, история багов — превращаются в векторные эмбеддинги и хранятся в векторном хранилище (pgvector или Qdrant). Агент делает семантический запрос: «правила ценообразования для стекинга купонов» — и получает только BL-PRICE-001, BL-PRICE-003 и ECL-2.3. Вместо 80+ правил — 3–5 релевантных.

Четыре конкретных значения:

- **Снижение стоимости токенов** — ~100 КБ контекст → ~5 КБ только нужного. Бюджет идёт на рассуждения, а не на загрузку ненужного.
- **Дедупликация тест-кейсов** — перед генерацией новый кейс проверяется на семантическое сходство с 1 546 существующими.
- **Поиск похожих багов** — перед созданием JIRA-тикета агент проверяет эмбеддинги прошлых багов. Никаких дублей, автосвязывание.
- **Умный test impact analysis** — эмбеддинг git diff → семантическое сопоставление с тест-кейсами → автовыбор scope регрессии без ручного маппинга.

Важный момент: Context7 — это уже RAG. Скил `/vc-docs` делает именно это — семантический поиск по документации Virto Commerce. Мы расширяем этот паттерн на всю внутреннюю базу знаний.
