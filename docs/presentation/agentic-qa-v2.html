The Problem We Solve

Traditional Automation vs. Agentic QA

❌ Traditional Test Automation

  • Requires Playwright/Selenium engineering skills
  • Breaks on every CSS class / selector change
  • High maintenance cost — devs, not QA, fix tests
  • Adding a test suite = writing hundreds of lines of code
  • Can't reason about business logic violations
  • No contextual understanding — only matches patterns

✅ Agentic QA

  • Plain English prompts — no coding required
  • Self-healing — agents find elements by context, not selectors
  • Low maintenance — update the prompt, not the test code
  • Adding a test suite = adding a CSV file
  • Agents apply business invariants (BL-* rules) to every test
  • Contextual reasoning: "this price is wrong because…"

Project Overview

What Is the Agentic QA System?

An LLM-powered testing platform where AI agents execute test cases through real browsers — no test code, no fragile selectors.
🧠

AI-Native

Tests are plain English prompts executed by Claude through MCP browser tools. The LLM reasons about what to click, what to verify, and what constitutes a bug.

🏗️

Modular Architecture

14 specialized agents, 36 CSV test suites, 18 methodology skills, and 12 shared knowledge files — all coordinated by an orchestrator.

⚙️

CI Pipeline Planned

GitHub Actions triggers daily smoke tests and weekly full regression. Docker container + Claude Agent SDK run suites headlessly with Teams notifications.

End-to-End Flow

/qa-test VCST-XXXX — From Ticket to Verdict

One command reads the JIRA ticket, dispatches the right agents, executes all acceptance criteria, files bugs with evidence, and closes the ticket.

Step 1: Read JIRA Ticket

Atlassian MCP fetches title, description, acceptance criteria · identifies feature scope: storefront / admin / API

Step 2: Dispatch Specialists

qa-frontend-expert (Chrome) → storefront ACs · qa-backend-expert (Edge) → API + admin ACs · run in parallel

Step 3: Execute Tests

Navigate to feature · verify each AC through real browser · capture screenshots + HAR · check console for errors

Step 4: File Bugs (if any)

Failed AC → bug report with P0–P3 severity · screenshots · HAR · BL-* rule reference · auto-created in JIRA

Step 5: Close the Ticket

All ACs pass → transition ticket to "Testing Complete" · link evidence · notify team. Bugs found → block release.

Future Architecture · Planned

RAG + Vector Search — Scalable Agent Memory

Replace static knowledge file loading with semantic retrieval — agents query a vector store and receive only what's relevant, cutting token waste and enabling test intelligence at scale.

The Problem: Static Context Loading

  • 12 knowledge files injected into every agent — ~100 KB per call
  • BL-* pricing rules loaded even when testing auth flows
  • 1,546 test cases in CSV — no semantic search across them
  • Bug deduplication is manual — agents can't check past reports
  • Knowledge grows → context bloat → higher cost, slower reasoning
Context7 is already RAG — /vc-docs does semantic retrieval over live VC documentation. The full RAG layer extends this pattern to all internal knowledge.

Value Delivered

Focused context → lower token cost

Agent receives 3–5 relevant BL-* rules instead of 80+. Less noise, sharper reasoning. Token budget goes further per run.

Test case deduplication

/qa-test-cases-generator queries the vector store before writing — surfaces semantically similar existing cases, prevents duplicate coverage.

Bug similarity search

Before filing a JIRA ticket, agent checks embeddings of past bug reports — eliminates duplicates and auto-links related issues.

Smart test impact analysis

Embed git diff → semantic match against test cases → auto-select the right regression scope. No manual suite mapping needed.
