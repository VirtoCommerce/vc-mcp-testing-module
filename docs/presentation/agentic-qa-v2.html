An LLM-powered testing platform where AI agents execute test cases through real browsers — no test code, no fragile selectors.

Knowledge File · Runner Contract

GraphQL xAPI — Schema Truth + Runner-Native Tests

Two knowledge files plus a canonical runner turn the xAPI GraphQL surface into deterministic, schema-validated CSV test cases — no custom JS scripts needed.
📐 knowledge/graphql-schema.md
Live introspection snapshot of the xAPI GraphQL schema — queries, mutations, input types, return types. Every agent consults this first to avoid invented field names.
Schema-update workflow: re-run introspection against {BACK_URL}/graphql when xAPI changes → regenerate graphql-schema.md → CI fails any test referencing dropped fields.
📋 knowledge/graphql-test-cases-runner.md
Canonical authoring contract: tag grammar, predicate shapes, capture chaining, common failure modes, checklist + worked example.

Tag Grammar (Steps · Assertions · Filters)

Steps: [AUTH] · [GQL-OP] · [GQL-VARS] · [GQL-EXEC] · [GQL-CAPTURE]
[REST-OP] · [REST-EXEC] · [REST-CAPTURE]
Assertions: [ERRORS] · [DATA] · [NULL] · [COUNT] · [VAR]
Filters: getByPath syntax · arithmetic · cross-path · OR-AND predicates

Concrete CSV Example — assertions in action

Steps:
[AUTH] storefront token for @td(SUPPORT_AGENT)
[GQL-OP] products
[GQL-VARS] filter: "category.subtree:@td(VIRTUAL_CATALOG_B2B.id)"
          first: 5
[GQL-EXEC]
[GQL-CAPTURE] $first = data.products.items[0]

Assertions:
[ERRORS] none
[NULL] data.products is not null
[COUNT] data.products.items >= 1
[DATA] $first.name matches /.+/
[DATA] $first.prices[0].actual.amount > 0
[VAR] $first.id resolves to @td(PRODUCT_LAPTOP.id)
🛠 scripts/graphql-runner.ts
Canonical runner — schema validation, var substitution, @td() resolution, evidence capture. Never write custom JS to execute GraphQL CSV cases.
npx tsx scripts/graphql-runner.ts --case <csv>:<ID>
Gold-standard suite: 050i-graphql-configurations.csv — full tag/predicate coverage, ready for new authors to copy

Knowledge File · Runtime Helpers

Live Test-Data Discovery — No Hardcoded IDs

Catalogs get reseeded. B2B orgs get re-created. Virtual-catalog roots migrate. Hardcoded IDs and SKUs rot fast. Live discovery resolves entities at runtime against the actual env.
📖 knowledge/live-discovery.md
Decision tree + recipes: when to pick {{VAR}} vs @td() vs live-discover vs random-data. Covers parallel-run isolation against the agent user pool and AGENT-TEST-* cleanup conventions.

Four Data Layers

LayerSourceUse for
{{VAR}}.env (33 vars)URLs, creds, env context
@td()aliases.json + CSVNamed entities you assert by name
live-discoverxAPI at runtime"First available product" — shape, not value
random-dataZero-dep generatorsUnique inputs (emails, comments, qtys)
🛠 scripts/lib/live-discover.ts
Typed xAPI primitives — catalog root, products by filter, addresses, cart state, active coupons. Returns a real entity from the current environment, never a hardcoded ID.
🎲 scripts/lib/random-data.ts
Zero-dep generators: emails, org names, SKUs, quantities, comments. Defaults to the AGENT-TEST-{date} prefix so /qa-seed-data teardown sweeps them safely.
Anti-pattern: hardcoding a product GUID or SKU in Test_Data. A literal that isn't {{VAR}} or @td() is a review failure (Dimension 6 — Data Validity).
# CSV-runner recipe — discover then capture

[GQL-CAPTURE] $firstProduct = data.products.items[0].id

End-to-End Flow

/qa-test VCST-XXXX — From Ticket to Verdict

One command reads the JIRA ticket, dispatches the right agents, executes all acceptance criteria, files bugs with evidence, and closes the ticket.

Step 1: Read JIRA Ticket

Atlassian MCP fetches title, description, acceptance criteria · identifies feature scope: storefront / admin / API

Step 2: Dispatch Specialists

qa-frontend-expert (Chrome) → storefront ACs · qa-backend-expert (Edge) → API + admin ACs · run in parallel

Step 3: Execute Tests

Navigate to feature · verify each AC through real browser · capture screenshots + HAR · check console for errors

Step 4: File Bugs (if any)

Failed AC → bug report with P0–P3 severity · screenshots · HAR · BL-* rule reference · auto-created in JIRA

Step 5: Close the Ticket

All ACs pass → transition ticket to "Testing Complete" · link evidence · notify team. Bugs found → block release.
