## Слайд 1: Title

Добро пожаловать! Сегодня поговорим про нашу систему агентного тестирования — это когда AI-агенты на базе Claude выполняют реальные тесты в настоящих браузерах, получая инструкции на обычном человеческом языке. Никакого кода, никаких скриптов — только промпты и умные агенты.

Ключевая идея: мы не заменяем тестировщиков, мы даём им суперспособности. Вместо написания `await page.click('.btn-submit')` тестировщик пишет «нажми кнопку оформления заказа» — а агент сам разбирается, где эта кнопка, даже если её CSS-класс поменялся вчера.

Технологический стек: Claude (Anthropic), Model Context Protocol (MCP), Playwright для браузеров, JIRA для тикетов, Figma для дизайн-сравнений. Всё это оркестрируется через Claude Code CLI прямо из IDE.

## Слайд 3: The Problem

Классическая автоматизация — это Playwright/Selenium-скрипты, которые ломаются при каждом изменении UI. Переименовали CSS-класс — тесты упали. У нас всё иначе.

**Конкретный пример:** В традиционном подходе логин выглядит так:
```javascript
await page.fill('#email', 'user@test.com');
await page.click('.btn-login');
```
Разработчик переименовал `.btn-login` в `.auth-submit` — тест упал. В агентном подходе инструкция звучит: «Войди в систему под тестовым пользователем». Агент сам находит поле email и кнопку входа по визуальному контексту и семантике страницы.

**Масштаб проблемы:** У нас 1 274 тест-кейса. Представьте, что каждый из них — это хрупкий скрипт с 10-15 селекторами. Один рефакторинг фронтенда = сотни упавших тестов. При агентном подходе рефакторинг фронтенда не ломает ни одного теста — потому что тестов-скриптов просто нет.

**Стратегия селекторов агента:** `data-testid` → `aria-label` → семантический HTML → визуальный контекст. Агент пробует в этом порядке и адаптируется.

## Слайд 5: Architecture Overview

У нас два режима работы, и это ключевое архитектурное решение.

**Интерактивный режим** (ежедневная работа):
- Инженер в IDE (VS Code / Cursor) запускает команду, например `/qa-test VCST-1234`
- Claude Code CLI активирует нужных агентов
- Агенты открывают до 3 браузеров параллельно через Playwright MCP
- Доступны все 9 MCP-серверов: браузеры, JIRA, Figma, DevTools, Postman
- Результаты: скриншоты, HAR-файлы, баг-репорты, обновления тикетов в JIRA

**CI-режим** (автоматический):
- Триггер: GitHub Actions (cron или manual)
- Среда: Docker-контейнер с `Playwright + Claude Agent SDK`
- Скрипт `ci/run-regression.ts` оркестрирует прогон
- Один headless Chromium (не 3 браузера — экономия ресурсов)
- Результаты: markdown-отчёт + JSON-саммари + Teams-уведомление

**Общий фундамент:** `config/test-suites.json` (манифест 36 сьютов), `regression/suites/` (CSV тест-кейсы), `.claude/agents/` (определения агентов). Написал один раз — работает в обоих режимах.

## Слайд 6: MCP Server Ecosystem

MCP (Model Context Protocol) — это протокол от Anthropic, который превращает Claude из текстового генератора в полноценного агента с «руками и глазами».

**Как это работает технически:** MCP-сервер — это процесс, который предоставляет Claude набор инструментов (tools) через JSON-RPC. Например, `playwright-chrome` даёт инструменты `browser_navigate`, `browser_click`, `browser_fill`, `browser_take_screenshot`. Claude вызывает эти инструменты как обычные функции.

**6 серверов на уровне проекта** (в `.mcp.json`):
- `playwright-chrome/firefox/edge` — три отдельных браузера, каждый со своим процессом, viewport 1920×1080, HAR-capture включён
- `postman` — API-тестирование через коллекции и окружения Postman
- `github` — работа с PR, issues, code search
- `context7` — поиск актуальной документации Virto Commerce в реальном времени

**3 сервера на уровне пользователя** (в настройках IDE):
- `Chrome DevTools` — доступ к консоли, network-запросам, performance tracing
- `Atlassian` — JIRA: создание багов, переходы по workflow, чтение ACs
- `Figma` — визуальное сравнение с макетами дизайнера

**Пример цепочки:** Агент открывает страницу каталога (Playwright) → проверяет консоль на ошибки (DevTools) → сравнивает layout с Figma (Figma MCP) → находит расхождение → создаёт баг в JIRA (Atlassian MCP) с прикреплённым скриншотом.

## Слайд 7: Agent Team Structure

Две команды, 11 агентов. Разделение не случайное — оно отражает реальные роли в QA-команде.

**QA-команда (7 агентов):**
- **qa-lead-orchestrator** (Sonnet) — координатор. Не тестирует сам! Читает тикеты, назначает специалистов, выносит вердикты. Sonnet — потому что ему нужна скорость принятия решений, а не глубокий анализ.
- **qa-frontend-expert** (Opus) — специалист по сторфронту. Проверяет checkout, каталог, корзину, мобильную адаптивность, кросс-браузер. Opus — потому что фронтенд-тестирование требует глубокого понимания UI-состояний.
- **qa-backend-expert** (Opus) — REST API, GraphQL xAPI (xCart, xCatalog, xOrder), Admin SPA, RBAC. Проверяет мутации, пагинацию, авторизацию.
- **qa-testing-expert** (Opus) — интерактивное тестирование: воспроизведение багов, Figma-сравнение, дебаг через DevTools.
- **ui-ux-expert** (Sonnet) — Storybook (55 компонентов по Atomic Design), WCAG 2.1 AA аудит, консистентность дизайн-системы.
- **test-management-specialist** (Sonnet) — планирование тестов, написание кейсов по 18 доменным чеклистам, трекинг покрытия.
- **regression-orchestrator** (Sonnet) — запуск 36 сьютов батчами по 3, ретраи, фоллбэк браузеров, консолидация отчётов.

**BA-команда (4 агента):** анализ системы, API-поверхность, user stories с BDD, документация.

**Почему Opus vs Sonnet?** Opus — мощнее, но дороже и медленнее. Используем его только там, где нужен глубокий анализ (тестирование). Sonnet — быстрее и дешевле, идеален для координации и документации.

## Слайд 8: Four-Layer Prompt Architecture

Каждый QA-агент определён в markdown-файле (`.claude/agents/*.md`) и его промпт строится в 4 слоя:

**Слой 1 — Бизнес-логика (Инварианты):**
Файл `knowledge/business-logic.md` — единый для ВСЕХ агентов. Содержит правила с уникальными ID:
- `BL-PRICE-001`: «Стекирование скидок не может превышать 100%»
- `BL-CART-004`: «Удаление последнего товара из корзины должно показать пустое состояние»
- `BL-ORD-001`: «После оплаты заказ переходит в статус Processing»
- `BL-AUTH-002`: «3 неудачных попытки входа → блокировка на 15 минут»

**Слой 2 — Доменные знания (Суждения):**
`platform-patterns.md` — как устроен Virto Commerce: «Checkout использует async-валидацию», «Admin SPA построен на VC-Shell (Angular)». `browser-quirks.md` — «Firefox иногда не триггерит blur при быстром Tab».

**Слой 3 — Техники (Инструментарий):**
Стратегия селекторов: `data-testid` → `aria-label` → semantic HTML. Порядок API-тестов: CRUD (Create → Read → Update → Delete). Правила захвата доказательств: HAR + скриншот + console log.

**Слой 4 — Ограничения (Границы):**
«Используй playwright-chrome», «Максимум 100 ходов», «Viewport 1920×1080», «При падении 3+ тестов — остановись и отчитайся».

**Зачем это нужно?** Когда frontend-эксперт проверяет checkout и видит, что итог не совпадает — он сверяется с BL-PRICE-001 и точно знает: это баг, а не фича.

## Слайд 9: Browser Isolation Strategy

Это жёсткое правило: два агента **никогда** не делят один браузер.

**Почему?** Playwright MCP-сервер — это один процесс с одним browser context. Если frontend-эксперт перейдёт на `/checkout`, а в этот момент backend-эксперт сделает `navigate('/admin')` — оба потеряют свой контекст. Куки, localStorage, текущая страница — всё перезаписывается.

**Схема назначения:**
| Агент | MCP-сервер | Порт |
|-------|-----------|------|
| qa-frontend-expert | playwright-chrome | Chromium |
| qa-testing-expert | playwright-firefox | Firefox |
| qa-backend-expert | playwright-edge | Edge (msedge) |
| ui-ux-expert | Chrome DevTools MCP | отдельный процесс |

**Цепочка фоллбэка:** Если chrome занят → пробуем firefox → пробуем edge. Это актуально при регрессии, когда оркестратор запускает несколько сьютов параллельно.

**Важный нюанс:** WebKit (Safari) на Windows не поддерживается Playwright — агент не должен даже пытаться его запустить. Конфиги это учитывают, фоллбэк сразу идёт на Edge.

**Каждый браузер настроен одинаково:** viewport 1920×1080, HAR-capture включён, видео при падении, изолированные контексты. Конфиги лежат в `config/mcp-playwright-*.config.json`.

## Слайд 10: Orchestration Pattern

Ключевой принцип: **лид-оркестратор координирует, но никогда сам не тестирует.** Это как тимлид, который распределяет задачи, но не пишет код.

**Алгоритм работы лида:**
1. **ANALYZE** — читает JIRA-тикет, парсит Acceptance Criteria, определяет scope
2. **DELEGATE** — назначает специалистов: фронтенд-изменения → frontend-expert, API → backend-expert, UI/UX → ui-ux-expert
3. **MONITOR** — отслеживает прогресс каждого агента
4. **RULES** — применяет бизнес-правила (BL-*) к результатам
5. **JUDGE** — выносит вердикт: APPROVED / CONDITIONAL / BLOCKED

**Пять типовых воркфлоу с примерами:**
- **Новая фича** (`/qa-test VCST-1234`): «Добавили BOPIS (Buy Online, Pick Up In Store)» → лид отправляет frontend-expert проверить UI, backend-expert — API, опционально ui-ux-expert — доступность
- **PR Review**: Backend проверяет API-контракты, frontend — UI-рендеринг, testing — интерактивно прокликивает
- **Релизное тестирование**: Лид вызывает regression-orchestrator, тот запускает батчи по 3 сьюта
- **Верификация багфикса**: testing-expert воспроизводит оригинальный баг → если не воспроизводится → test-management проверяет по чеклисту смежные сценарии

## Слайд 11: Regression Pipeline

Регрессионный пайплайн — это 9 шагов от манифеста до вердикта.

**Детально по шагам:**
1. `regression-orchestrator` читает `config/test-suites.json` — 36 сьютов с метаданными
2. Резолвит выборку: `smoke` → [01], `critical` → [01,06,08,14], `sprint` → [26 сьютов], `full` → все 36
3. Генерирует run ID: `REG-2026-03-05-0900`
4. Сортирует по приоритету — P0 (Smoke, Payment, Security, Platform API) идут первыми
5. Запускает батчами по 3 — ровно по числу браузерных слотов
6. Каждый суб-агент получает шаблон `docs/prompts/test-runner-agent.md` с подставленными параметрами: `{{SUITE_ID}}`, `{{BROWSER_SERVER}}`, `{{ENVIRONMENT_URL}}`
7. При падении — до 2 ретраев с фоллбэком на другой браузер
8. Собирает JSON-результаты от всех суб-агентов
9. Применяет quality gates → финальный вердикт

**Пример:** `sprint` selection → 26 сьютов → 9 батчей по 3 → ~600 тест-кейсов. Каждый батч занимает 15-25 мин. Общее время ~2-3 часа вместо дней ручного прогона.

**Группы выборки:** smoke (12 тестов, ежедневно), critical (120, P0-гейт), sprint (600, перед релизом спринта), full (1 274, перед продом), отдельно frontend (492) и backend (782).

## Слайд 12: Quality Gates

Quality gates — это не рекомендации, а жёсткие автоматические правила. Ни один агент не может их обойти.

**Три уровня гейтов:**
- **Smoke** (ежедневный): 100% pass rate, 0 P0, 0 P1. Это бинарно: GO или NO-GO. Если хоть один P0-тест упал — деплоя не будет.
- **Sprint** (перед релизом спринта): ≥95% pass rate, 0 P0, P1 допускаются. Если pass rate 94% — BLOCKED, даже если все P0 прошли.
- **Full Release** (перед продакшеном): ≥95% pass rate, 0 P0, ≤3 P1. Четвёртый P1-баг → BLOCKED.

**Три вердикта:**
- **APPROVED** — все гейты пройдены. Деплой безопасен.
- **CONDITIONAL** — основные гейты пройдены, но есть P2/P3 issues. Деплоим, но трекаем follow-up тикеты.
- **BLOCKED** — есть P0/P1 или low pass rate. Деплоя не будет, и никакое давление по дедлайнам это не изменит.

**Пример из практики:** Sprint regression показал 96% pass rate, 0 P0, но 2 P1 (сломанный фильтр в каталоге + ошибка пагинации в заказах). Вердикт: CONDITIONAL — деплоим, но оба P1 идут в следующий спринт как P0.

## Слайд 13: CI/CD Pipeline

CI-пайплайн — это полностью автоматизированный прогон тестов в Docker.

**Техническая схема:**
```
GitHub Actions (cron / manual trigger)
  → docker build -t vc-regression -f ci/Dockerfile .
  → docker run --shm-size=2gb --env-file .env
      -e ANTHROPIC_API_KEY=...
      -e SUITE_SELECTION=smoke
      -e MAX_BUDGET_USD=5.0
  → ci/run-regression.ts (TypeScript, Claude Agent SDK)
  → Headless Chromium (один браузер, не три)
  → reports/regression/ci-YYYY-MM-DD/
  → ci/notify-teams.ts → Teams Adaptive Card
```

**Расписание:**
- **Пн-Пт 6:00 UTC** — smoke (Suite 01, 12 тестов, бюджет $5). Если упал — команда узнаёт до начала рабочего дня.
- **Воскресенье 2:00 UTC** — full regression (все 36 сьютов, бюджет $80). Результат готов к понедельнику.
- **Manual** — любая выборка, любое окружение, любой бюджет через `workflow_dispatch`.

**Бюджет-менеджмент:** Общий бюджет делится между сьютами с минимальным полом $2 на сьют. Это важно — если агент «упрётся» в сложный тест и потратит бюджет, он должен корректно завершиться, а не зависнуть.

**Teams-уведомления:** Adaptive Card с таблицей: сьют → статус (pass/fail) → pass rate → количество тестов → время. Плюс общий вердикт и ссылка на полный отчёт.

**История:** 90-дневный rolling window в `reports/regression/history.json` — можно отслеживать тренды.

## Слайд 14: Commands & Skills Ecosystem

Команды и навыки — это два типа инструментов, и важно понимать разницу.

**Команды (9 штук)** — это action-oriented триггеры. Они запускают полноценные воркфлоу с агентами:
- `/qa-smoke` → лид-оркестратор → 2 параллельных агента → GO/NO-GO за 15 мин
- `/qa-test VCST-1234` → лид парсит тикет → назначает специалистов → собирает результаты
- `/qa-regression sprint` → regression-orchestrator → 26 сьютов батчами по 3
- `/qa-bug «Корзина показывает неправильную сумму»` → testing-expert воспроизводит → создаёт баг в JIRA
- `/qa-exploratory checkout` → guided сессия с heuristics (SFDPOT, CRISP)

**Навыки (18 штук)** — это методологические библиотеки. Они не запускают агентов, а подключают знания:
- `/qa-checklist checkout` → загружает 158 пунктов по 18 доменам (Auth, Catalog, Cart, Checkout, Payment, B2B, ...)
- `/qa-risk sprint` → матрица рисков 5×5, приоритизация тестов по severity × probability
- `/qa-test-design VCST-1234` → техники: Equivalence Partitioning, BVA, Decision Tables, State Transitions
- `/vc-docs dynamic properties` → Context7 ищет актуальную документацию Virto Commerce
- `/qa-accessibility /catalog` → WCAG 2.1 AA аудит по принципам POUR

**Пример связки:** `/qa-test VCST-1234` (команда) вызывает лида, лид использует `/qa-checklist` (навык) чтобы понять скоуп, затем `/qa-risk` (навык) чтобы определить глубину тестирования, затем делегирует агентам.

### Приложение к слайду 14: Подробный справочник навыков (Skills)

**Что такое Skills в Claude Code?**

Skills (навыки) — это markdown-файлы в `.claude/skills/`, которые расширяют возможности агента структурированными знаниями и методологиями. Каждый навык — это `SKILL.md` с YAML-фронтматтером + опциональные supporting-файлы с детальной справочной информацией.

**Отличие от команд:** Команда (`/qa-smoke`) запускает рабочий процесс с агентами и побочными эффектами (открывает браузеры, создаёт тикеты). Навык (`/qa-checklist`) подключает знания в контекст агента — это чистый input, никаких побочных эффектов.

**Как это устроено технически:**
- Claude Code сканирует `.claude/skills/` и показывает их в меню `/`
- YAML-поле `description` — описание в меню (с тегом `[Category]`)
- YAML-поле `argument-hint` — подсказка для аргументов
- YAML-поле `disable-model-invocation: true` — запрет авто-вызова (без этого Claude может вызвать навык самостоятельно)
- Supporting-файлы читаются on-demand: агент загружает только те, что нужны для текущей задачи

#### Категория 1: VC Knowledge (4 навыка) — auto-invocable

Знания о платформе Virto Commerce. Могут вызываться автоматически (без `disable-model-invocation`), т.к. read-only и без побочных эффектов.

**`/vc-docs` — Документация Virto Commerce**
- **Что делает:** Ищет актуальную документацию через Context7 MCP (`/virtocommerce/vc-docs`). Архитектура, модули, API, деплоймент.
- **Пример:** `/vc-docs dynamic properties` → как работают динамические свойства в VC
- **Плюсы:** Всегда актуальная информация (живой запрос к документации), не нужно держать копии доков в репо
- **Минусы:** Зависит от доступности Context7 MCP-сервера; при сетевых проблемах навык не работает; результаты могут быть неполными для очень специфичных тем

**`/vc-module` — Модули и маппинг на тест-сьюты**
- **Что делает:** Показывает зависимости модуля, его API-поверхность и маппинг на конкретный regression suite
- **Пример:** `/vc-module catalog` → «Catalog module → Suite 17 (Catalog Admin), зависит от search, pricing»
- **Supporting:** `module-suite-map.md` — таблица «модуль → сьют → приоритет → тест-кейсы»
- **Плюсы:** Быстрый ответ «что тестировать, если обновили модуль X»; связывает бизнес-знания с тестовыми артефактами
- **Минусы:** Маппинг нужно обновлять вручную при добавлении новых сьютов; не покрывает cross-module зависимости автоматически

**`/vc-api` — xAPI и REST API справочник**
- **Что делает:** Справочник по GraphQL xAPI (xCart, xCatalog, xOrder, xCMS) и REST API. Примеры запросов, мутаций, авторизации.
- **Пример:** `/vc-api xCart` → мутации addItem, removeItem, changeQuantity с примерами GraphQL-запросов
- **Supporting:** `xapi-query-ref.md` — готовые запросы с переменными
- **Плюсы:** Агент сразу получает готовые примеры запросов для тестирования; не нужно каждый раз искать в документации
- **Минусы:** Справочник отражает конкретную версию API; при мажорных обновлениях xAPI нужно обновлять вручную

**`/vc-frontend` — Справочник по сторфронту**
- **Что делает:** Полный sitemap, структура навигации, типы продуктов, структура аккаунта, тестовые данные
- **Пример:** `/vc-frontend checkout` → URL `/cart`, `/checkout/shipping`, `/checkout/payment`, `/checkout/review`
- **Supporting:** `sitemap.md` — полная карта сайта с URL-ами и описаниями (обновлена март 2026)
- **Плюсы:** Агент точно знает, по каким URL ходить; не тратит время на exploration; включает тестовые данные (логины, организации)
- **Минусы:** Привязан к конкретному деплойменту; URL могут отличаться для кастомных тем; при рефакторинге URL нужно обновлять sitemap

#### Категория 2: Testing (6 навыков) — manual invocation

Техники тестирования. Каждый делегирует работу конкретному агенту-специалисту.

**`/qa-storybook` — Визуальная регрессия Storybook**
- **Что делает:** Тестирование 55 компонентов по Atomic Design (atoms, molecules, organisms). Визуальное сравнение, responsive breakpoints (mobile 375px, tablet 768px, desktop 1920px).
- **Делегирует:** ui-ux-expert
- **Пример:** `/qa-storybook Button` → скриншоты всех состояний (default, hover, active, disabled, loading) × 3 viewport
- **Supporting:** `visual-regression-testing.md`, `responsive-component-testing.md`
- **Плюсы:** Покомпонентное тестирование в изоляции от всего приложения; ловит визуальные поломки до прода; стандартизированные breakpoints
- **Минусы:** Требует работающий Storybook-сервер (`STORYBOOK_URL`); baseline-скриншоты пока не персистентные (ad-hoc); не ловит баги интеграции компонентов на реальных страницах

**`/qa-accessibility` — WCAG 2.1 AA аудит**
- **Что делает:** Аудит доступности по принципам POUR (Perceivable, Operable, Understandable, Robust). Проверяет контраст, фокус-менеджмент, alt-тексты, ARIA-атрибуты, keyboard-навигацию.
- **Делегирует:** ui-ux-expert
- **Пример:** `/qa-accessibility /catalog` → проверяет страницу каталога: контраст текста (4.5:1 минимум), tab order, screen reader compatibility
- **Supporting:** `wcag-accessibility-checklist.md`
- **Плюсы:** Систематический подход (не случайные проверки); привязка к стандарту WCAG 2.1 AA; находит проблемы, которые разработчики обычно пропускают
- **Минусы:** Автоматизированный аудит ловит ~30-40% проблем доступности — полный аудит требует ручного тестирования с assistive technologies; ложноположительные на динамических компонентах

**`/qa-design` — Консистентность дизайн-системы**
- **Что делает:** Проверяет соответствие UI дизайн-системе VC + UX-эвристики Якоба Нильсена (10 heuristics). Типографика, цвета, отступы, иконки.
- **Делегирует:** ui-ux-expert
- **Пример:** `/qa-design checkout` → проверяет: единообразие кнопок, spacing grid (8px), шрифты, цвета по палитре
- **Supporting:** `design-system-consistency.md`, `ux-heuristic-evaluation.md`
- **Плюсы:** Ловит UI-drift (когда разные разработчики используют разные отступы/шрифты); формализует UX-критерии вместо субъективного «выглядит нормально»
- **Минусы:** Без доступа к Figma-макетам проверяет только внутреннюю консистентность, а не соответствие дизайну; эвристики — это guidelines, а не абсолютные правила

**`/qa-plan` — Планирование тестов из E2E-каталога**
- **Что делает:** Компонует тест-план из каталога 105 E2E-сценариев. Подбирает релевантные сценарии по фиче, домену или тикету.
- **Делегирует:** test-management-specialist
- **Пример:** `/qa-plan checkout` → выбирает 12 E2E-сценариев: guest checkout, registered, multi-item, promo code, address validation, payment flows...
- **Supporting:** `e2e-scenario-catalog.md` — 105 сценариев с приоритетами
- **Плюсы:** Не нужно каждый раз придумывать тесты с нуля; покрытие основано на реальных user flows; приоритеты уже расставлены
- **Минусы:** Каталог ограничен 105 сценариями — edge cases нужно добавлять вручную; не генерирует тест-кейсы автоматически (только план)

**`/qa-checklist` — Доменные чеклисты**
- **Что делает:** 18 предустановленных доменных чеклистов + Bug Fix Verification checklist. Всего 158 пунктов. Можно создать новый чеклист для нового домена.
- **Делегирует:** test-management-specialist
- **Домены (18):** Auth, Catalog, Cart, Checkout, Payment, Orders, B2B/Company, Search, Navigation, Account, Addresses, Wishlists, i18n, Performance, Security, Admin SPA, CMS, SEO
- **Пример:** `/qa-checklist checkout` → 14 пунктов: валидация адреса, выбор доставки, промокод, итого, переход к оплате, пустые поля, дубликаты заказов...
- **Пример:** `/qa-checklist new "push messages"` → генерирует новый чеклист по шаблону из `checklist-creation-guide.md`
- **Supporting:** `domain-checklists.md` (158 пунктов), `checklist-creation-guide.md` (как создавать новые)
- **Плюсы:** Готовый «мозговой штурм» за секунду; каждый пункт маппится на минимум один тест-кейс; покрывает edge cases, которые легко забыть
- **Минусы:** Чеклисты — это напоминание, а не замена тест-дизайна; для полного покрытия нужно комбинировать с `/qa-test-design`; нужно обновлять при добавлении новой функциональности

**`/qa-api` — REST API и GraphQL xAPI тестирование**
- **Что делает:** Готовые тест-кейсы для API-тестирования: CRUD-операции, авторизация, пагинация, error handling, GraphQL-мутации.
- **Делегирует:** qa-backend-expert
- **Пример:** `/qa-api xCatalog` → тест-кейсы: query products, filter by category, search by SKU, check pagination, verify price calculation
- **Supporting:** `test-cases-api-graphql.md`
- **Плюсы:** Готовые тест-сценарии для API — не нужно писать с нуля; покрывает и REST, и GraphQL; включает негативные сценарии (401, 403, 404, 422)
- **Минусы:** Фиксированный набор тест-кейсов; при добавлении новых endpoints нужно расширять supporting-файл; не тестирует performance/load

#### Категория 3: QA Methodology (8 навыков) — cross-team practices

Методологические навыки. Не привязаны к конкретному агенту — используются всей командой.

**`/qa-process` — Жизненный цикл по ISTQB (корневой навык)**
- **Что делает:** 7 фаз тестирования с формальными entry/exit criteria: Plan → Analyze → Design → Implement → Execute → Report → Close
- **Примеры:**
  - `/qa-process` → полный обзор + таблица «фаза → навыки → артефакты»
  - `/qa-process analyze VCST-1234` → парсит AC тикета, выводит нумерованные test conditions
  - `/qa-process implement` → 12-пунктовый pre-execution checklist (окружение готово? данные есть? MCP-серверы работают?)
  - `/qa-process close sprint-42` → 15-пунктовый close checklist + шаблон ретроспективы
- **Supporting:** `test-process-lifecycle.md`
- **Плюсы:** Единая структура для всего QA-процесса; формальные критерии «когда входить/выходить из фазы»; маппинг фаз на конкретные навыки и агентов
- **Минусы:** ISTQB-модель может быть избыточной для простых задач; формальность замедляет work при hotfixes; entry criteria могут блокировать прогресс, если не все данные доступны

**`/qa-test-design` — Техники тест-дизайна**
- **Что делает:** Систематическая деривация тест-кейсов: Equivalence Partitioning (EP), Boundary Value Analysis (BVA), Decision Tables, State Transitions, Pairwise, Error Guessing
- **Пример:** `/qa-test-design VCST-1234` → анализирует AC → определяет подходящие техники → генерирует тест-кейсы
- **Пример:** «Поле `quantity` принимает 1-999» → BVA: тестируем 0, 1, 2, 998, 999, 1000
- **Supporting:** `test-design-techniques.md`
- **Плюсы:** Минимальный набор тестов с максимальным покрытием (научный подход вместо «проверим что в голову придёт»); воспроизводимо
- **Минусы:** Требует чётких спецификаций для EP/BVA; Decision Tables экспоненциально растут при множестве условий; State Transitions сложно строить для UI с множеством состояний

**`/qa-risk` — Оценка рисков**
- **Что делает:** Матрица 5×5 (Severity × Probability), классификация severity/priority, распределение глубины тестирования по уровню риска
- **Пример:** `/qa-risk sprint` → оценивает все фичи спринта, ранжирует по риску, рекомендует: «Payment flow — exhaustive, Search filter — standard, Footer link — minimal»
- **Supporting:** `risk-prioritization-framework.md`
- **Плюсы:** Обоснованное распределение усилий вместо «тестируем всё одинаково»; помогает при ограниченном времени — фокус на высокорисковых областях
- **Минусы:** Субъективность оценки probability; исторические данные по дефектам пока не используются автоматически; риск не учитывает бизнес-контекст (например, промо-сезон)

**`/qa-investigate` — Расследование багов (5 фаз)**
- **Что делает:** Reproduce → Isolate → Analyze → Document → Verify. Включает 8 паттернов типичных багов Virto Commerce (P1-P8)
- **Пример:** `/qa-investigate "Корзина показывает неправильную сумму"` → воспроизводит → изолирует (только с промокодом?) → анализирует (BL-PRICE-003 нарушен) → документирует
- **Supporting:** `bug-investigation-flow.md`
- **Плюсы:** Структурированный подход вместо хаотичного «потыкаем и посмотрим»; паттерны P1-P8 ускоряют распознавание типичных проблем; результат — полный баг-репорт
- **Минусы:** 5 фаз могут быть излишни для очевидных багов; требует доступ к нескольким MCP-серверам одновременно; не все баги воспроизводимы в MCP-браузере (race conditions, timing)

**`/qa-evidence` — Захват доказательств**
- **Что делает:** Политика захвата (что, когда, как), 3 уровня отчётов (compact/detailed/signoff), стандартные пути вывода, шаблоны sign-off
- **Пример:** `/qa-evidence signoff` → генерирует таблицу sign-off для frontend + backend с чеклистами
- **Supporting:** `evidence-capture-policy.md`, `output-paths.md`, `sign-off-templates.md`
- **Плюсы:** Единообразие отчётов по всей команде; чёткие правила «что прикладывать к багу» (HAR, скриншот, console log); готовые sign-off шаблоны для Teams
- **Минусы:** Может создавать overhead для мелких багов; HAR-файлы могут быть большими; sign-off процесс — дополнительный шаг

**`/qa-defect` — Управление дефектами**
- **Что делает:** Полный JIRA Bug Workflow (16 статусов), triage-процесс, классификация (severity/priority/type), верификация, метрики дефектов
- **Примеры:**
  - `/qa-defect triage VCST-1234` → оценивает severity, priority, назначает owner
  - `/qa-defect verify VCST-1234` → проверяет фикс, прогоняет regression в смежных областях
  - `/qa-defect workflow` → показывает 16 статусов JIRA и переходы между ними
  - `/qa-defect metrics` → дефектная плотность, reopen rate, time to fix
- **Supporting:** `defect-lifecycle-workflow.md`, `defect-report-templates.md`
- **Плюсы:** Стандартизированный workflow (все баги проходят одинаковый путь); автоматическая классификация; шаблоны отчётов — не нужно каждый раз писать с нуля
- **Минусы:** 16 статусов могут быть избыточны для маленьких команд; зависит от Atlassian MCP для JIRA-операций; тriage может не учитывать бизнес-контекст без input от PO

**`/qa-metrics` — Метрики качества и гейты**
- **Что делает:** Каталог метрик (pass rate, defect density, DRE — Defect Removal Efficiency, coverage, escape rate), определение quality gates, автоматическое применение гейтов
- **Пример:** `/qa-metrics gates` → показывает пороги: Smoke 100%, Sprint ≥95%, Full ≥95% + P0=0 + P1≤3
- **Пример:** `/qa-metrics trends` → анализирует history.json и показывает тренды за 90 дней
- **Supporting:** `quality-metrics-catalog.md`, `quality-gates.md`
- **Плюсы:** Объективные критерии go/no-go (не «мне кажется, что ок»); regression-orchestrator автоматически применяет гейты; тренды помогают видеть деградацию качества
- **Минусы:** Метрики могут быть «gamed» (увеличить pass rate, снизив приоритет тестов); DRE требует данных из прода (escape bugs); пороги могут быть слишком строгими для MVP-фич

**`/qa-sbtm` — Session-Based Test Management (exploratory testing)**
- **Что делает:** Структурированное исследовательское тестирование: SBTM-чартеры, эвристики CRISP/SFDPOT, тестовые туры (Tourist, Guidebook, Money, Landmark), дебриф-шаблон
- **Пример:** `/qa-sbtm checkout` → генерирует чартер: «Explore checkout flow, focusing on edge cases with multiple payment methods. Time: 60 min. Heuristics: SFDPOT (Structure, Function, Data, Platform, Operations, Time)»
- **Supporting:** `session-based-testing.md`
- **Плюсы:** Структура для исследовательского тестирования (а не бесцельное «потыкаем»); чартеры создают фокус; дебриф фиксирует находки для будущих сессий; хорошо ловит баги, которые скриптовые тесты пропускают
- **Минусы:** Результаты зависят от опыта тестировщика/агента; сложнее измерить покрытие; 60-минутные сессии могут быть дорогими (API cost)

#### Общая оценка Skills-архитектуры

**Плюсы системы навыков в целом:**
- Знания отделены от агентов — можно обновить чеклист без изменения агента
- Модульность — навыки комбинируются (risk + test-design + checklist)
- Версионирование — всё в git, history видна
- Расширяемость — новый навык = новая папка с SKILL.md
- Граф зависимостей создаёт целостную методологию, а не набор разрозненных инструментов

**Минусы / ограничения:**
- Supporting-файлы нужно обновлять вручную при изменениях в платформе
- Навыки увеличивают контекстное окно агента — при загрузке нескольких навыков одновременно можно превысить лимит
- Нет автоматической валидации актуальности (файл может устареть, и никто не заметит)
- Формальная методология (ISTQB, SBTM) может быть overhead для простых задач — нужен здравый смысл при выборе глубины

#### Как агенты узнают, когда использовать навыки и когда читать знания?

Это ключевой вопрос архитектуры: как 11 агентов «понимают», когда загрузить навык, а когда обратиться к файлу знаний? Ответ — три механизма маршрутизации.

---

**Механизм 1: Прямые ссылки в промпте агента (hardcoded)**

Каждый агент определён в markdown-файле (`.claude/agents/*.md`). В этом файле есть прямые ссылки на конкретные knowledge-файлы и навыки — агент загружает их при каждом запуске.

Пример из `qa-frontend-expert.md`:
```
> **Reference:** `.claude/agents/knowledge/business-logic.md` — testable business invariants
> **Reference:** `.claude/agents/knowledge/platform-patterns.md` — storefront patterns
> **Reference:** `.claude/agents/knowledge/browser-quirks.md` — browser limitations
> **Reference:** `.claude/agents/knowledge/debugging-signals.md` — console error patterns
> **Reference:** `.claude/agents/knowledge/performance-thresholds.md` — LCP, CLS targets
```

Это **обязательные знания** — агент всегда имеет к ним доступ. Как учебник, который всегда лежит на столе.

Пример из `test-management-specialist.md`:
```
> **Reference:** `.claude/agents/knowledge/business-logic.md` — бизнес-инварианты для покрытия тест-кейсами
+ Memory Model (16 long-term reference files):
  - e2e-scenario-catalog.md — 105 E2E-сценариев по 18 доменам
  - domain-checklists.md — 18 доменных чеклистов, 158 пунктов
  - test-design-techniques.md — EP, BVA, Decision Tables, State Transitions
  - risk-prioritization-framework.md — матрица 5×5, глубина тестирования
  - test-process-lifecycle.md — ISTQB 7-фазный жизненный цикл
  - output-paths.md — пути сохранения артефактов
  - module-suite-map.md — маппинг модулей на тест-сьюты
  - catalog.md, store-settings.md, white-labeling.md — доменные знания
```

Тест-менеджер всегда знает бизнес-правила, каталог сценариев и чеклисты — это нужно ему для каждого тест-плана.

---

**Механизм 2: Таблица «ситуация → навык» в промпте агента (conditional)**

В промпте каждого агента есть таблица-маршрутизатор: «Когда ты в такой ситуации — загрузи такой навык».

Пример из `qa-frontend-expert.md`:
```
| Ситуация                           | Навык → файл                                      |
|------------------------------------|----------------------------------------------------|
| Начало тест-сессии                  | /qa-evidence → evidence-capture-policy.md           |
| Вывод тест-кейсов из JIRA          | /qa-test-design → test-design-techniques.md         |
| Приоритизация глубины тестирования  | /qa-risk → risk-prioritization-framework.md         |
| Исследовательское тестирование     | /qa-sbtm → session-based-testing.md                 |
| Завершение (sign-off)              | /qa-evidence → sign-off-templates.md                |
```

Пример из `test-management-specialist.md`:
```
| Ситуация                                | Навык → файл                                      |
|-----------------------------------------|----------------------------------------------------|
```

Это **условные знания** — загружаются on-demand при определённых условиях. Как справочник, который берёшь с полки, когда нужен.

**Механизм 3: Автоматический вызов (auto-invocable skills)**

4 навыка из категории `vc-knowledge` НЕ имеют `disable-model-invocation: true` в YAML. Это значит, что Claude может вызвать их самостоятельно, без явной инструкции.

Как это работает на практике:
1. Агент тестирует каталог и сталкивается с непонятным поведением фильтров
2. Claude «видит» навык `/vc-docs` в доступных tools
3. Автоматически вызывает `/vc-docs catalog filters` — и получает актуальную документацию из Context7
4. Использует эту информацию для принятия решения: баг или фича?

Аналогично:
- Агент встречает незнакомый URL → автоматически вызывает `/vc-frontend` → получает sitemap
- Агент видит GraphQL-ошибку → автоматически вызывает `/vc-api` → получает правильный формат запроса
- Агент тестирует модуль → автоматически вызывает `/vc-module` → узнаёт зависимости и связанный сьют

Остальные 14 навыков заблокированы от авто-вызова (`disable-model-invocation: true`), потому что:
- Testing-навыки делегируют агентам (побочные эффекты: открытие браузеров)
- Methodology-навыки тяжеловесные (загружают большие supporting-файлы, раздувают контекст)
- Их должен вызывать человек или лид-оркестратор осознанно

## Слайд 15: Knowledge Sharing Layer

Слой знаний — это 8 markdown-файлов в `.claude/agents/knowledge/`, которые все QA-агенты используют как общий справочник.

**Самый важный файл — `business-logic.md`:**
Содержит тестируемые бизнес-инварианты с уникальными ID. Примеры:
- `BL-PRICE-001`: Стекирование скидок не может превышать 100%
- `BL-PRICE-003`: Цена товара × количество + доставка + налог = итого в корзине
- `BL-CART-004`: Пустая корзина → показать empty state с CTA «Continue Shopping»
- `BL-ORD-001`: Оплата прошла → статус заказа = Processing
- `BL-ORD-002`: Отмена заказа → возврат товара на склад (inventory reversal)
- `BL-AUTH-002`: 3 неудачных попытки входа → блокировка на 15 минут
- `BL-B2B-001`: Пользователь видит только каталог своей организации

**Как агент использует:** Проверяя checkout, frontend-expert смотрит BL-PRICE-003 и валидирует, что `subtotal + shipping + tax = total`. Если не совпадает — это точно баг, а не «так задумано».

**Остальные файлы:**
- `platform-patterns.md` — «Admin SPA использует blade-навигацию», «xAPI мутации требуют CSRF-token»
- `browser-quirks.md` — «Firefox: `blur` event не триггерится при быстром Tab», «Edge: download dialog блокирует execution»
- `debugging-signals.md` — «Console error `ChunkLoadError` = проблема с code splitting», «401 на `/connect/token` = истёкший refresh token»
- `performance-thresholds.md` — LCP < 2.5s, CLS < 0.1, API response < 3s, time to interactive < 5s

## Слайд 16: Skill Dependency Graph

Навыки — это не изолированные инструменты, они образуют граф зависимостей. Это важно для понимания методологической целостности системы.

**Корень: `qa-process`** — жизненный цикл тестирования по ISTQB (7 фаз: Plan → Analyze → Design → Implement → Execute → Report → Close). Каждая фаза имеет entry/exit criteria.

**Ветвь тест-дизайна:**
`qa-test-design` (техники: EP, BVA, Decision Tables, State Transitions, Pairwise)
  → `qa-plan` (композиция тест-сьюта из каталога 105 E2E-сценариев)
  → `qa-checklist` (18 доменных чеклистов, 158 пунктов)

**Пример:** Тестируем новый платёжный метод. `qa-test-design` определяет техники (BVA для сумм, State Transitions для workflow платежа). `qa-plan` подбирает релевантные E2E-сценарии. `qa-checklist` загружает Payment-домен с 12 пунктами.

**Ветвь рисков:**
`qa-risk` (матрица 5×5: severity × probability)
  → `qa-sbtm` (Session-Based Test Management: чартеры, CRISP/SFDPOT heuristics, туры)

**Ветвь дефектов (замкнутый цикл):**
`qa-investigate` (5 фаз расследования бага)
  → `qa-defect` (JIRA Bug Workflow: 16 статусов, triage, классификация)
    → `qa-metrics` (DRE — Defect Removal Efficiency, escape rate, defect density)

**Цикл обучения:** нашли баг → зафиксировали в JIRA → оценили риск → провели SBTM-сессию в смежной области → обновили метрики → скорректировали приоритеты. Система становится умнее с каждым багом.

## Слайд 17: Agent Routing — Skills vs Knowledge

Это «клей» всей архитектуры — как 11 агентов и 18 навыков работают вместе без хаоса.

**Пять механизмов маршрутизации** (показаны на схеме слайда):

1. **Hardcoded references** — прямые ссылки в промпте агента, загружаются всегда. Пример: `test-management-specialist` загружает 16 reference-файлов при каждом запуске — от `business-logic.md` до `catalog.md`. Это его «память».

2. **Conditional routing table** — таблица «ситуация → навык» прямо в промпте. Когда тест-менеджер пишет тест-кейсы — он загружает `/qa-plan`, когда проверяет покрытие — `/qa-checklist`, когда оценивает риски — `/qa-risk`. У него 13 строк в таблице — больше всех в команде.

3. **Auto-invocable skills** — 4 навыка `vc-knowledge` не имеют `disable-model-invocation: true`. Claude вызывает их автоматически: встретил незнакомый URL → `/vc-frontend` → получил sitemap. Без явной команды.

4. **ROUTING.md** — системный файл-маршрутизатор с decision tree: «Я хочу X → используй Y». Помогает и людям (выбрать команду), и агентам (выбрать навык).

5. **Agent→Skill Map** — формальная таблица в README навыков. Явная привязка: кто что использует. Позволяет верифицировать, что агент не «забыл» про нужный навык.

**Ключевой инсайт:** Знания (knowledge) = память агента (всегда доступны). Навыки (skills) = справочники (загружаются по ситуации). Это как разница между тем, что врач помнит наизусть, и справочником, который он открывает для конкретного диагноза.

**Почему это работает:** Агент не угадывает — у него детерминированные правила. Hardcoded references гарантируют базовый контекст. Conditional tables дают on-demand глубину. Auto-invocable skills закрывают неожиданные ситуации. Три уровня — от жёсткого к гибкому.

## Слайд 18: A Smoke Test in Action

Вот как выглядит ежедневный ритуал — от одной команды до финального вердикта.

**Шаг за шагом:**
1. **Инженер набирает `/qa-smoke`** в IDE (VS Code / Cursor)
2. **qa-lead-orchestrator** активируется, читает `config/test-suites.json`, находит Suite 01 (Smoke)
3. **Генерирует run ID:** `SMOKE-2026-03-05-0900`
4. **Запускает два параллельных трека:**
   - **Track A:** `qa-frontend-expert` в `playwright-chrome` — 12 P0-тестов: логин, каталог, корзина, checkout, поиск, навигация
   - **Track B:** `qa-backend-expert` в `playwright-edge` — health check Admin SPA: логин в админку, проверка dashboard, базовые CRUD
5. **Оба агента выполняют тесты** через реальные браузеры — кликают, заполняют формы, проверяют результаты
6. **Оркестратор собирает результаты** и применяет правила:
   - Все тесты прошли + админка жива = **GO**
   - Checkout упал = **NO-GO** (это revenue-critical flow)
   - 3+ любых падения = **NO-GO**
7. **Записывает `smoke-report.md`** и выдаёт вердикт: `"GO — Safe to deploy"`

**Время:** ~15 минут от команды до ответа. Для сравнения: ручной smoke-тест занимает 1-2 часа.

**Что в отчёте:** таблица тестов с pass/fail, время каждого теста, скриншоты для failed, общий вердикт, рекомендации.

## Слайд 19: Benefits & Results

Главные выигрыши — и почему это не просто «модная технология», а реальное улучшение процесса.

**Self-healing (самовосстановление):**
UI поменялся — тесты не сломались. Агент находит элементы по контексту, а не по CSS-селекторам. Пример: кнопка «Add to Cart» переехала из `.product-actions` в `.product-cta` — традиционный тест упал, агентный — нет.

**Нулевой порог входа:**
QA-инженер пишет промпты на обычном языке: «Проверь, что после добавления товара в корзину badge показывает правильное количество». Не нужно знать JavaScript, Playwright, CSS-селекторы.

**Параллельное выполнение:**
3 агента в 3 браузерах одновременно. Sprint regression (600 тестов) за 2-3 часа вместо дней.

**JIRA из IDE:**
Прочитать тикет, создать баг, перевести статус — не переключаясь из IDE. `/qa-bug` создаёт полный баг-репорт с STR, expected/actual, скриншотами.

**Встроенная методология:**
18 навыков покрывают ISTQB (процесс), SBTM (exploratory), WCAG 2.1 AA (accessibility), risk-based testing. Это не просто тесты — это полноценный QA-процесс.

**Автоматический сбор доказательств:**
HAR-файлы (полный network trace), скриншоты (desktop + mobile), console logs, API traces. Всё прикрепляется к баг-репорту автоматически.

**Модульная архитектура:**
Новый сьют = CSV + запись в manifest. Новый навык = markdown-файл. Новый агент = markdown-файл. Масштабирование без кода.

## Слайд 20: Future Roadmap

Текущая система — крепкий фундамент, но есть куда расти.

**Мультибраузерный CI:**
Сейчас CI использует один headless Chromium. Планируем добавить Firefox и Edge в Docker — те же 3 параллельных слота, что и в интерактивном режиме. Это даст реальное кросс-браузерное тестирование в CI.

**Визуальная регрессия:**
Персистентные скриншоты компонентов Storybook как baseline. При каждом прогоне — pixel-by-pixel сравнение. Сейчас `/qa-storybook` делает это ad-hoc, нужен автоматический pipeline.

**BrowserStack интеграция:**
Мобильное тестирование на реальных устройствах: iOS Safari, Android Chrome. Это критично для B2C-сценариев. Уже есть env-переменные `BROWSERSTACK_USERNAME` и `BROWSERSTACK_ACCESS_KEY`.

**AI-генерация тест-кейсов:**
Агент читает Acceptance Criteria из JIRA-тикета и автоматически генерирует тест-кейсы в CSV-формате. Тестировщик ревьюит и утверждает. Экономия ~50% времени на написание тестов.

**Sprint coverage reports:**
Автоматический маппинг: какие файлы изменились в спринте → какие тест-сьюты затронуты → какие тесты нужно прогнать. Это переход от «прогоняем всё» к «прогоняем только то, что могло сломаться».

**Самообновляющиеся сьюты:**
Регрессия, которая адаптируется к изменениям в коде — добавляет новые тест-кейсы, помечает устаревшие, перераспределяет приоритеты.
